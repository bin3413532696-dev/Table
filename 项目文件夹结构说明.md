# 项目文件夹结构说明

**最后更新**: 2026-05-05

## 1. 文档目的

本文档用于明确 `Table` 项目当前目录结构的真实职责，并给出后续整理约束。

当前原则是：

- 先完成架构收口
- 再做目录瘦身
- 不在当前阶段进行高风险的大规模文件搬迁

## 2. 当前根目录结构

```text
Table/
├─ src/                     前端 React 应用
├─ server/                  后端 API 与异步投影运行时
├─ prisma/                  Prisma schema 与 migrations
├─ scripts/                 冒烟、端到端、辅助脚本
├─ data/                    开发期调试数据与历史文件产物
├─ dist/                    前端构建产物
├─ dist-server/             后端构建产物
├─ node_modules/            依赖目录
├─ .env                     本地环境配置
├─ package.json             前后端统一脚本入口
├─ webpack.config.js        前端开发与构建配置
├─ 架构重构实施方案.md       重构总方案与阶段状态
├─ 交接文档.md              当前接手入口
└─ 项目文件夹结构说明.md     当前目录职责说明
```

## 3. 当前目录职责

### 3.1 `src/`

前端展示层、缓存层、交互层所在目录。

当前一级目录职责：

- `agent/`
  - 前端智能体上下文、工具注册、执行编排
- `components/`
  - 通用组件与布局组件
- `contexts/`
  - React Context 容器
- `core/`
  - 基础类型、错误、事件、通用能力
- `db/`
  - 前端业务数据访问封装
- `hooks/`
  - React hooks
- `kb/`
  - 知识库前端访问层、查询封装、数据组装
- `lib/`
  - 初始化、同步、网络能力封装
- `pages/`
  - 页面路由入口
- `store/`
  - 本地缓存与过渡状态容器
- `styles/`
  - 样式资源
- `sync/`
  - 前端同步与刷新策略

### 3.2 `server/`

后端权威服务目录，当前是项目真实业务写入入口。

当前结构建议理解为：

- `server/src/db/`
  - Prisma client 与数据库接入
- `server/src/modules/tasks/`
  - 任务模块
- `server/src/modules/finance/`
  - 财务模块
- `server/src/modules/knowledge/`
  - 知识库模块
- `server/src/modules/projection/`
  - outbox 与知识投影运行时
- `server/src/modules/maintenance/`
  - reset、snapshot、seed 等开发运维能力
- `server/src/modules/health/`
  - 健康检查
- `server/src/shared/`
  - 共享工具、通用类型、基础设施

### 3.3 `prisma/`

数据库结构定义目录。

当前职责：

- `schema.prisma`
  - 所有已落地表结构定义
- `migrations/`
  - 数据库演进历史

### 3.4 `scripts/`

项目脚本目录。

当前应只放以下类型内容：

- 冒烟脚本
- 端到端验证脚本
- 结构校验脚本
- 开发辅助脚本

不建议在这里继续堆放临时试验脚本。

### 3.5 `data/`

当前目录不再是业务主数据权威源。

现在的定位是：

- 历史开发期文件产物保留位
- 调试与对照数据保留位
- 少量知识库文件级诊断参考位

约束：

- 不再把 `tasks`、`finance` 的真实写入依赖建立在这里
- 不再围绕 `data/` 继续扩展新业务逻辑
- 后续若保留，应逐步收敛为 `fixtures/`、`snapshots/` 或 `debug/` 语义

### 3.6 `dist/` 与 `dist-server/`

构建产物目录。

约束：

- 不承载手工维护代码
- 不写架构文档
- 不提交业务逻辑补丁到构建产物中

## 4. 当前目录整理结论

当前项目目录虽然仍有一些过渡痕迹，但主干已经清晰：

- `src/` 是前端层
- `server/` 是后端权威层
- `prisma/` 是数据库层
- `scripts/` 是验证与辅助层
- `data/` 是遗留调试层，不再是权威层

因此，本阶段“整理项目文件夹结构”的正确做法不是大规模搬目录，而是先固定职责边界，避免继续把新代码写进错误位置。

## 5. 后续整理约束

从现在开始，建议遵循以下约束：

1. 新业务写入逻辑只能进入 `server/src/modules/*`
2. 新数据库结构只能通过 `prisma/schema.prisma` 与 migration 演进
3. 前端不再新增直接落文件的主流程
4. 新的知识刷新逻辑优先收敛到 `src/sync/`，不要再分散到页面层
5. 新的智能体后端化能力应直接新增 `server/src/modules/agent/`
6. 搜索能力应新增 `server/src/modules/search/`
7. 用户体系与鉴权能力应新增 `server/src/modules/auth/`

## 6. 推荐的下一阶段目录收敛

当当前重构主线稳定后，建议做一次目录级收敛，目标结构如下：

```text
Table/
├─ docs/
│  ├─ architecture/
│  ├─ handoff/
│  └─ operations/
├─ src/
├─ server/
│  └─ src/
│     └─ modules/
│        ├─ tasks/
│        ├─ finance/
│        ├─ knowledge/
│        │  └─ ontology/
│        ├─ projection/
│        ├─ agent/
│        ├─ search/
│        ├─ auth/
│        └─ maintenance/
├─ prisma/
├─ scripts/
│  ├─ smoke/
│  ├─ e2e/
│  └─ ops/
├─ fixtures/
└─ logs/
```

说明：

- `docs/`
  - 后续统一承载方案、交接、运维文档
- `fixtures/`
  - 若继续保留样本数据，应从 `data/` 中拆出
- `logs/`
  - 当前根目录下的运行日志后续应统一收口到这里

## 7. 当前不建议立即做的事

当前不建议立即执行以下目录动作：

- 大规模移动 `src/` 或 `server/` 业务代码
- 直接删除 `data/` 目录
- 在未补回引用的情况下移动根目录文档
- 一次性重命名多个核心目录

原因：

- 当前项目仍处于架构重构连续阶段
- 目录大搬迁会放大联调、脚本、文档引用和运行路径风险
- 现阶段收益最高的是边界收口，不是形式上的目录美化

## 8. 接手建议

接手者应按以下方式理解当前目录：

- 看业务权威逻辑，优先看 `server/`
- 看数据库结构，优先看 `prisma/`
- 看前端如何接 API，优先看 `src/db/`、`src/kb/`、`src/sync/`
- 看知识投影，优先看 `server/src/modules/projection/`
- 看开发验证，优先看 `scripts/`

如果后续继续推进大改，应先完成：

1. 用户体系
2. 智能体服务端化
3. 搜索与索引层
4. ontology 约束与编辑接口

之后再做目录搬迁，成本最低，收益最高。
