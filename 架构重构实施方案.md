# 架构重构实施方案

## 1. 文档目的

本文档用于定义 `Table` 项目的大规模架构重构方案，目标不是局部修补当前实现，而是将项目从“前端本地存储驱动的单机工作台”升级为“可持续扩展的智能工作台平台底座”。

## 1.1 当前重构前提

当前项目中的用户数据主要为历史测试数据，可直接舍弃，不作为迁移保留目标。

这意味着本次重构默认采用：

- 新底座重建
- 旧系统逻辑参考
- 非保守型数据兼容策略

不默认要求：

- 保留旧 `localStorage` 数据
- 保留旧 `data/*.json` 数据
- 新旧系统双写
- 历史主键、历史快照的长期兼容

本文档覆盖：

- 重构目标与设计原则
- 目标架构与模块边界
- 数据模型与存储方案
- 智能体执行模型
- 同步、搜索、知识投影方案
- 前后端改造路径
- 分阶段迁移计划
- 风险、成本与验收标准

---

## 2. 现状问题概述

当前项目的核心问题不在于功能少，而在于底层职责混合，导致系统上限偏低。

### 2.1 当前实现的主要结构

- 前端页面直接驱动业务数据读写
- 任务和财务通过内存 Store + `localStorage` 持久化
- 知识库通过另一套内存数据集 + `localStorage` 快照维护
- Webpack Dev Server 通过 `/api/sync-data`、`/api/load-data` 将数据读写到 `data/*.json`
- 智能体直接依赖前端当前状态和本地工具调用链路

### 2.2 当前结构的主要问题

#### 2.2.1 权威源不唯一

当前系统同时存在以下“像权威源”的数据位置：

- 前端内存状态
- `localStorage`
- 知识库内存数据集
- `data/*.json` 文件

这会带来：

- 谁是真实数据不清楚
- 启动恢复和冲突处理复杂
- 多模块写入一致性差

#### 2.2.2 写入链路过长

一次业务写入可能触发：

- 更新内存 Store
- 覆盖 `localStorage`
- 更新知识库映射
- 触发同步到服务端
- 落盘到 JSON 文件

这类链式写入没有真正的事务边界，失败后容易形成中间态。

#### 2.2.3 智能体执行不具备产品级可恢复性

当前智能体更接近“前端工具调用器”，而不是正式的执行系统，缺少：

- 统一 run 状态
- 工具调用审计
- 用户确认后的恢复机制
- 服务端持久化执行链路

#### 2.2.4 知识库和主业务双写耦合过深

任务、财务写入会同步修改知识库，这导致：

- 主业务依赖知识库状态
- 知识层失败会污染业务写路径
- 后续投影与重建能力弱

#### 2.2.5 同步与扩展能力偏弱

当前方案适合单机开发期，不适合：

- 多端同步
- 多用户协作
- 冲突处理
- 审计
- 服务端权限控制

---

## 3. 重构目标

本次重构目标分为四层。

### 3.1 一致性目标

- 所有业务真相收敛到单一权威源
- 所有主写操作必须通过后端服务
- 主数据和派生数据职责拆分

### 3.2 工程目标

- 模块化单体，明确边界，避免继续堆叠页面逻辑
- 前后端职责分离
- 可做独立测试、回放、审计

### 3.3 产品目标

- 保持本地优先体验
- 支持智能体可靠执行
- 支持知识增强检索与跨模块分析

### 3.4 上限目标

- 支持多业务模块扩展
- 支持多端同步
- 支持多用户与权限
- 支持搜索、知识图谱、自动化、智能体能力持续增强

### 3.5 迁移目标调整

由于历史测试数据可舍弃，本次重构不以“平滑保留旧数据”为优先目标，而以“尽快建立正确的新架构底座”为优先目标。

因此目标调整为：

- 优先建立新后端权威源
- 优先建立新数据库 schema
- 优先建立新 API 与新执行链路
- 使用 seed 数据替代历史测试数据

## 4. 设计原则

### 4.1 单一权威源

业务真相只能有一个。

- 前端不是权威源
- `localStorage` 不是权威源
- JSON 文件不是权威源
- 后端数据库是唯一权威源

### 4.2 主数据与派生数据分离

主数据：

- 任务
- 财务
- 用户设置

派生数据：

- 知识图谱实体与关系
- 搜索索引
- 智能体摘要、记忆、工具执行记录
- dashboard 聚合数据

派生数据失败不应阻塞主业务提交。

### 4.3 所有写入统一收口

所有业务写入都必须经过：

- API Controller
- Domain Service
- Repository
- 事务提交

前端不再直接修改底层存储。

### 4.4 异步投影优先

任务、财务到知识图谱、搜索、智能体上下文的同步全部作为异步投影处理，不做强事务双写。

### 4.5 缓存与真相分离

- 本地缓存只负责性能、离线体验、草稿与队列
- 后端数据库负责最终一致性与权威状态

### 4.6 可渐进迁移

重构必须能分阶段推进，不要求一次替换所有模块。

### 4.7 舍弃历史测试数据优先于兼容旧结构

如果旧结构与新架构目标冲突，应优先采用更合理的新结构，而不是为了兼容测试数据继续保留旧设计。

这包括但不限于：

- ID 生成规则
- 表结构字段命名
- 知识库拆表方式
- 智能体执行持久化方式
- 配置存储位置

## 5. 目标架构

## 5.1 总体架构

```text
Web App
  ├─ UI / Route / Components
  ├─ Query Cache
  ├─ IndexedDB Cache
  └─ Offline Mutation Queue

API Server (Modular Monolith)
  ├─ Tasks Module
  ├─ Finance Module
  ├─ Knowledge Module
  ├─ Agent Module
  ├─ Settings/Auth Module
  ├─ Sync Module
  ├─ Search Module
  └─ Audit Module

Primary Database
  └─ PostgreSQL

Async Layer
  ├─ Outbox / Job Queue
  ├─ Knowledge Projection Worker
  ├─ Search Index Worker
  └─ Agent Execution Worker
```

## 5.2 为什么采用模块化单体

不直接采用微服务，原因如下：

- 当前项目业务边界不少，但规模还不足以支撑服务拆分收益
- 任务、财务、知识库、智能体之间仍然需要较强事务与协调能力
- 单体内部模块化更利于快速落地和统一治理
- 后期如果业务量继续放大，仍可按模块边界演进拆分

结论：

- 当前阶段最佳选择是模块化单体
- 不是最终不能拆，而是先用更低复杂度换更高成功率

## 6. 后端模块设计

## 6.1 模块列表

建议拆分为以下业务模块：

- `tasks`
- `finance`
- `knowledge`
- `agent`
- `settings`
- `auth`
- `sync`
- `search`
- `audit`
- `shared`

## 6.2 模块内部分层

每个模块统一采用以下分层：

- `controller`：处理 HTTP / WebSocket / SSE 请求
- `service`：承载业务规则、事务、领域逻辑
- `repository`：数据库访问
- `events`：领域事件定义
- `jobs`：异步任务与投影逻辑
- `dto`：输入输出协议
- `types`：模块内核心类型

## 6.3 模块协作原则

- 模块之间只通过 service、事件、公共接口协作
- 不允许页面级逻辑越过 service 直接操作 repository
- 不允许知识模块反向修改任务/财务主数据
- 不允许智能体直接碰数据库表，必须经领域工具层

## 8. 前端存储设计

## 8.1 职责变化

前端的职责应从“主存储写入者”改为“展示层 + 缓存层 + 离线体验层”。

## 8.2 前端存储分层

### 8.2.1 `localStorage`

只保留轻量配置：

- `theme`
- 局部 UI 偏好
- 极小型非关键配置

不再承担主业务数据存储。

### 8.2.2 `IndexedDB`

用于：

- 列表缓存
- 详情缓存
- 表单草稿
- 离线 mutation 队列
- 最近 agent 会话缓存

### 8.2.3 Query Cache

建议引入 `TanStack Query`：

- 管理远端状态
- 管理缓存失效
- 管理 optimistic update
- 管理请求重试与后台刷新

## 8.3 前端原则

- 页面不直接操作本地主数据
- 页面只发起 query / mutation
- 本地离线能力只作为增强，不参与真相裁决

## 9. 业务写入与领域服务设计

## 9.1 基本写路径

统一写路径：

`UI -> API -> Controller -> Service -> Repository -> DB Transaction`

然后再：

`Service -> emit domain event -> outbox job -> projection worker`

## 9.2 任务模块示例

创建任务流程：

1. 前端调用 `POST /api/tasks`
2. `TaskController` 校验输入
3. `TaskService.createTask()` 开启事务
4. 写入 `tasks`
5. 写审计日志
6. 写 `outbox_jobs`，发布 `task.created`
7. 事务提交
8. worker 消费 `task.created`
9. 更新知识图谱投影、搜索索引、智能体上下文

## 9.3 财务模块示例

流程与任务模块相同，只是投影目标不同：

- 更新财务主数据
- 异步生成财务知识实体
- 更新分类统计投影
- 更新相关搜索索引

## 10. 知识库重构方案

## 10.1 角色重新定义

知识库不是主业务真相，而是：

- 语义层
- 关联层
- 检索层
- 智能体上下文层

## 10.2 数据来源

知识库数据来源分两类：

### 10.2.1 主动维护的数据

- 用户直接创建实体
- 用户直接创建文档
- 用户直接创建断言/关系

### 10.2.2 派生投影的数据

- 任务投影为知识实体
- 财务记录投影为知识实体
- 跨模块关系自动生成

## 10.3 知识库更新原则

- 主业务修改只发事件，不直接强耦合修改知识表
- 知识库可重建
- 所有投影逻辑必须幂等

## 10.4 重建能力

需要提供知识重建任务：

- 全量重建任务投影
- 全量重建财务投影
- 重建断言和关系
- 重建搜索索引

这是确保长期可维护的关键能力。

## 11. 搜索与检索设计

## 11.1 第一阶段

采用 `PostgreSQL Full-Text Search`：

- 任务搜索
- 财务搜索
- 知识实体与文档搜索

## 11.2 第二阶段

增加向量检索：

- 文档 embedding
- 实体 embedding
- 混合检索：结构化过滤 + 语义召回

技术方向：

- `pgvector`
- 向量异步生成 worker

## 11.3 检索服务职责

统一搜索服务负责：

- 多模块搜索
- 结果聚合
- 权限过滤
- 语义检索
- 智能体上下文召回

## 12. 智能体重构方案

## 12.1 重构目标

将智能体从“前端工具调用器”升级为“后端受控执行系统”。

## 12.2 核心原则

- 智能体不直接碰浏览器本地状态
- 智能体不直接碰数据库
- 智能体只能调用受控工具层
- 工具层映射到领域 command/query

## 12.3 智能体执行链路

建议标准链路：

1. 用户提交消息
2. 创建 `agent_run`
3. 持久化消息
4. 模型生成工具计划
5. 持久化 `tool_execution`
6. 若需确认，进入 `pending_confirmation`
7. 用户确认后继续同一 run
8. 执行完成后写结果与审计

## 12.4 智能体工具层

工具应映射为服务端能力：

- `query_tasks`
- `create_task`
- `update_task`
- `delete_task`
- `query_finance`
- `add_finance_record`
- `search_knowledge`
- `upsert_knowledge_entity`
- `get_settings`
- `update_settings`

## 12.5 智能体能力上限

完成重构后，智能体可支持：

- 长流程执行
- 中断恢复
- 确认恢复
- 审计回放
- 权限限制
- 工具风险分级

## 13. 同步设计

## 13.1 同步目标

- 支持多端
- 支持离线
- 支持增量更新
- 支持基础冲突检测

## 13.2 同步原则

不再整包覆盖同步，采用版本化增量同步。

## 13.3 基础机制

每条记录带：

- `version`
- `updated_at`
- `deleted_at`

每次 mutation：

- 客户端带上当前版本
- 服务端检查是否陈旧
- 成功则返回最新实体
- 冲突返回 `409`

## 13.4 实时更新

建议采用：

- WebSocket，或
- SSE

推送内容：

- 资源类型
- 资源 ID
- 变更版本
- 变更类型

## 13.5 离线支持

前端保留 `offline mutation queue`：

- 离线时先记录 mutation
- 恢复网络后顺序重放
- 若遇冲突则提示用户处理

---

## 14. 审计与安全设计

## 14.1 审计范围

以下操作必须审计：

- 任务新增/修改/删除
- 财务新增/修改/删除
- 知识库手动编辑
- 智能体工具执行
- 设置变更

## 14.2 风险分级

工具按风险分级：

- 只读：自动执行
- 低风险写：可配置自动确认
- 高风险写：必须用户确认

## 14.3 配置安全

敏感配置不应继续裸存浏览器本地。

长期目标：

- API Key 存服务端加密字段
- PIN 哈希存数据库
- 重要配置读写受权限控制

## 16. 分阶段实施计划

## 16.1 Phase 0：准备期

目标：

- 冻结当前存储设计继续扩散
- 补齐现有关键测试
- 识别迁移边界

工作项：

- 盘点所有 `localStorage` key
- 盘点所有智能体工具写路径
- 盘点知识库与任务/财务耦合点

## 附录 A：PostgreSQL 表结构设计

## A.1 设计目标

本附录用于细化 PostgreSQL 数据库设计，作为本方案中“后端权威数据源”的落地结构。

目标：

- 为任务、财务、知识库、智能体执行提供稳定主存储
- 为审计、同步、投影、检索提供扩展基础
- 在不保留历史测试数据的前提下，直接按合理结构建新库

## A.2 数据库设计原则

### A.2.1 PostgreSQL 是唯一权威源

不再把以下位置作为主真相：

- 浏览器 `localStorage`
- 前端内存 Store
- `data/*.json`
- Webpack Dev Server 文件接口

### A.2.2 主数据与派生数据分离

主数据：

- 任务
- 财务记录
- 用户设置
- 智能体运行记录

派生数据：

- 知识图谱投影
- 搜索索引
- 统计聚合
- 向量检索

### A.2.3 软删除优先

核心业务表默认使用：

- `deleted_at timestamptz null`

### A.2.4 显式版本控制

所有核心表建议具备：

- `version integer not null default 1`

每次更新：

- `updated_at` 更新
- `version = version + 1`

### A.2.5 时间字段统一

统一使用：

- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

### A.2.6 JSON 字段只放低频灵活结构

- 核心可筛选字段做普通列
- 低频元数据放 `jsonb`

## A.3 Schema 规划建议

第一阶段建议全部放在 `public` schema，降低迁移复杂度。

后续若复杂度上升，可再按领域拆分：

- `core`
- `knowledge`
- `agent`
- `audit`
- `infra`

## A.4 通用字段规范

所有核心表建议包含以下字段子集：

- `id uuid primary key`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- `deleted_at timestamptz null`
- `version integer not null default 1`

## A.14 旧数据处理策略

当前决策：

- 历史测试数据可直接舍弃
- 不做正式历史业务数据迁移
- 旧数据只保留为结构参考、测试夹具或 seed 样本来源

建议动作：

1. 备份当前仓库和 `data/` 目录
2. 导出一份现有样本数据
3. 从样本中提取字段与边界情况
4. 基于新 schema 编写 seed 数据，而不是迁移脚本
5. 用新 seed 数据初始化开发和测试环境
