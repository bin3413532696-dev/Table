## P1 - 中等问题（待修复）

### P1-1: 知识库搜索使用 `ilike` 全表扫描，无全文索引

**文件**: `server/src/modules/knowledge/repository.ts:146-189`

**问题**: 搜索功能通过 `ilike '%query%'` 实现，在 `title` 和 `content` 字段上做模糊匹配。这两个字段没有索引，且 `ilike` 无法利用 B-Tree 索引（即使有索引也低效）。

**影响**: 知识库数据量增长后，查询性能线性下降，大数据量时可能数秒才能返回。

**修复建议**:
1. 使用 PostgreSQL `tsvector` + GIN 索引做全文搜索
2. 或引入 Elasticsearch / Meilisearch 等专用搜索引擎
3. 至少添加 `LIMIT` 和分页，避免一次返回过多结果

---

### P1-2: `listAllTags` 加载所有笔记仅取标签，浪费 IO

**文件**: `server/src/modules/knowledge/repository.ts:201-217`

**问题**: 为了获取所有使用过的标签，先 `findMany` 加载该用户的所有笔记（只选 `tagsJson`），然后在内存中展开去重。

**影响**: 用户有 1000 条笔记时，需要传输 1000 个 JSON 数组到应用层再处理。

**修复建议**: 使用原生 SQL 直接查询去重标签：
```sql
SELECT DISTINCT jsonb_array_elements_text(tags_json)
FROM knowledge_notes
WHERE user_id = ? AND deleted_at IS NULL
```

### P1-3: 乐观锁 version 字段未检查并发冲突

**文件**: `server/src/modules/providers/service.ts:220-235`（同样存在于 tasks、finance 模块的 service 层错误处理）

**问题**:
- `updateProviderForCurrentUser` 在更新时 `where` 条件中不包含 `version` 检查
- Tasks / Finance 的 repository 层已在 `where` 中添加了 `version` 条件，但 service 层未捕获 `P2025`（Record to update not found）错误并转换为友好的版本冲突提示

**影响**: 并发更新场景下（如多端同步），Providers 数据可能被静默覆盖；Tasks/Finance 更新失败时返回 404 而非明确的版本冲突错误。

**修复建议**:
1. Providers: 在 `updateProviderSchema` 中添加 `version: z.number().optional()`，service 层在 `where` 中检查 `version`
2. Tasks/Finance/Providers: 捕获 Prisma `P2025` 错误，返回 `409 Conflict` 与明确的版本冲突消息

### P1-4: API 响应格式不一致

**文件**: 多个后端路由文件

**问题**:
| 模块 | 列表格式 | 单条格式 |
|------|---------|---------|
| Tasks | `{ items, total, source }` | `{ data: task, source }` |
| Finance | `{ items, total, source }` | `{ data: record, source }` |
| Knowledge | `{ items, total, source }` | `{ data: note, source }` |
| Auth | `{ data: { items, total } }` | `{ data: { user, auth } }` |
| Agent | `{ items, total, source }` | run 对象直接返回 |

**影响**: 前端需要针对每个模块写不同的解析逻辑，容易引入 bug。

**修复建议**: 统一使用标准信封格式，如 `{ data, meta?, error? }`。

---

### P1-5: 前端 API 缺乏运行时类型校验

**文件**: `src/lib/agentApi.ts:96`, `src/db/index.ts:85` 等

**问题**: 大量使用 `response.json() as Promise<T>`，信任后端返回的数据结构完全正确。

**影响**: 后端接口升级或出现异常响应时，前端静默使用错误类型的数据，导致难以调试的运行时错误。

**修复建议**: 使用 Zod 对 API 响应做运行时校验，失败时抛出明确的 `ApiSchemaError`。

---

### P1-6: `trimMessagesHistory` 时间复杂度 O(n²)

**文件**: `src/agent/AgentContext.tsx:49-69`

**问题**: 在 while 循环中使用 `trimmed.shift()` 移除数组头部元素。JavaScript 数组 `shift()` 是 O(n) 操作，总时间复杂度 O(n²)。

**影响**: 长对话历史（如 1000 条消息）时，消息裁剪操作会导致明显的 UI 卡顿。

**修复建议**: 使用 `slice()` 找到合适的起始索引，一次性截取子数组，或改用链表结构。

---

### P1-7: `AgentContext` 的 `loadHistoryRun` 不重置处理状态

**文件**: `src/agent/AgentContext.tsx:106-113`

**问题**: `LOAD_HISTORY` action 没有重置 `isProcessing` 状态。如果在 Agent 正在流式响应时用户切换历史记录，`isProcessing` 可能仍为 `true`。

**影响**: UI 状态不一致，可能导致"停止"按钮无法使用或发送按钮被禁用。

**修复建议**: `LOAD_HISTORY` 时同时设置 `isProcessing: false`。

---

### P1-8: Cookie 缺少 `Secure` 标记

**文件**: `server/src/modules/auth/routes.ts:248`, `267`

**问题**: `Set-Cookie` 包含 `HttpOnly; SameSite=Lax`，但没有 `Secure`。

**影响**: 生产环境如果使用 HTTPS，没有 `Secure` 标记的 cookie 可能在中间人攻击中被窃取。

**修复建议**: 根据环境变量动态添加 `Secure` 标记：`process.env.NODE_ENV === 'production' ? 'Secure;' : ''`。

---

### P1-9: 开发服务器 `allowedHosts: 'all'` 存在 DNS Rebinding 风险

**文件**: `webpack.config.js:57`

**问题**: `devServer: { allowedHosts: 'all' }` 允许任意 Host 头访问开发服务器。

**影响**: 攻击者通过 DNS Rebinding 可以让用户的浏览器向本地 dev server 发起跨域请求，窃取开发数据或执行未授权操作。

**修复建议**: 限制为 `['localhost', '127.0.0.1']`。

---

### P1-10: Prisma 连接池未配置

**文件**: `server/src/db/client.ts:10-12`

**问题**: `new PrismaClient({ log: ['error', 'warn'] })` 没有显式配置连接池参数。

**影响**: 高并发场景下（如大量 Agent SSE 连接），可能快速耗尽数据库连接，导致请求堆积和超时。

**修复建议**:
```typescript
new PrismaClient({
  log: ['error', 'warn'],
  connectionLimit: 20, // 或根据部署环境配置
  poolTimeout: 10,
})
```

### P1-11: 缺少 API 速率限制

**文件**: `server/src/app.ts`

**问题**: Fastify 实例没有注册任何 rate limit 插件。`@fastify/rate-limit` 未安装。

**影响**:
1. Agent 流式接口可能被恶意频繁调用，耗尽 Provider API 额度和服务器资源
2. `/api/maintenance/reset` 可被重复调用造成 DoS
3. 搜索接口可能被刷

**修复建议**: 安装并配置 `@fastify/rate-limit`，对敏感接口（agent、auth、maintenance）设置更严格的限制。

---

### P1-12: `baselineReadyUsers` / `baselineInFlight` 不支持多进程

**文件**: `server/src/shared/auth.ts:16-17`

**问题**: 这两个变量是进程内内存状态。如果使用 PM2 cluster 模式或多容器部署，每个进程独立维护，导致基线补齐逻辑重复执行。

**影响**: 新用户首次请求可能触发多次样本数据初始化，产生重复的任务/财务/笔记记录。

**修复建议**: 使用数据库级别的分布式锁（如 `pg_advisory_lock`）或 Redis 锁替代内存状态。

---

## 总结

项目整体架构已从单机应用成功迁移到前后端分离，核心边界（前端 UI 层、后端业务层、PostgreSQL 权威层）较为清晰。

**本次已修复 9 个严重（P0）问题**，主要集中在：
1. **安全漏洞**: 维护接口过度权限、日志泄露敏感信息、PIN 哈希不安全
2. **功能缺陷**: SyncEngine 数据加载后未持久化、知识库硬删除与软删除策略矛盾、SSE 解析和错误处理不完整

**剩余待修复 14 个中等（P1）问题和 8 个低（P2）问题**，重点方向：
1. **性能**: 知识库搜索全文索引、标签查询优化、消息裁剪算法优化
2. **并发安全**: Providers 乐观锁补全、基线补齐多进程支持
3. **工程化**: API 响应格式统一、运行时类型校验、ESLint/Prettier 配置
4. **安全加固**: Cookie Secure 标记、API 速率限制、开发服务器 Host 限制
