## 1. 文档定位

本文档面向研发团队，重点描述：

- 代码级模块拆分
- 核心时序与调用链
- 关键数据表职责

## 2. 研发视角下的总体结构

当前项目本质上是一个：

**React 前端 + Fastify 后端 + Prisma/PostgreSQL + 异步投影运行时**

的单仓库平台底座。

### 2.1 研发分层

```text
src/
  页面层
  前端状态层
  数据访问门面
  同步引擎
  知识访问层

server/src/
  App 装配层
  鉴权与用户上下文
  领域模块层
  共享基础设施层
  异步投影运行时

prisma/
  数据模型定义
  migration 历史
```

### 2.2 当前最重要的架构约束

1. 前端不再是权威写入端
2. 后端模块是业务规则唯一入口
3. 知识库主权威不再是本地文件或 localStorage
4. 任务/财务到知识库不能回退为页面层双写
5. 智能体运行状态不能回退为纯前端状态机

## 3. 前端模块架构

## 3.1 `src/db/index.ts`

这是前端业务数据访问门面，不是数据库层。

职责：

- 封装任务、财务、维护模块 API
- 做响应错误映射
- 刷新前端缓存
- 在任务/财务变更后调度知识投影刷新

当前实现要点：

- `requestApi()` 是统一请求入口
- 已修复“无 body 请求仍强制设置 `Content-Type: application/json`”问题
- `financeDB` 和 `taskDB` 都在写后调用 `scheduleKnowledgeRefresh()`

设计含义：

- 页面不应直接 `fetch('/api/tasks')`
- 新增任务/财务相关业务，应优先扩展此门面，而不是把 HTTP 细节散落到页面

## 3.2 `src/sync/SyncEngine.ts`

这是知识库同步的唯一协调器。

职责：

- 防抖调度知识同步
- 全量推送知识数据集到服务端
- 从服务端加载知识权威数据集
- 获取知识元数据摘要
- 对外暴露同步状态

当前关键常量定义在：

- [src/sync/config.ts](D:\project\Table\src\sync\config.ts)

当前接口约束：

- 写入：`PUT /api/knowledge/dataset`
- 读取：`GET /api/knowledge/dataset`
- 元数据：`GET /api/knowledge/metadata`

关键结论：

- 当前同步层只服务知识库
- 不要把任务/财务同步能力再混进 `SyncEngine`

## 3.3 `src/kb/sync.ts`

这是知识库前端 API facade。

职责：

- 封装知识实体、文档、断言、关系、Ontology 的远程 CRUD
- 在成功写入后调用 `refreshKnowledgeDataset()`
- 将页面的知识操作统一映射到服务端

当前特点：

- 大多数写入接口采用 `POST/PATCH/DELETE`
- 所有写后刷新都回到数据集接口
- 已修复无 body 请求错误请求头问题

这层不应承担：

- 本地权威快照恢复
- 文件级知识调试逻辑
- 与任务/财务直接耦合的双写逻辑

## 3.4 `src/agent/AgentContext.tsx`

这是智能体前端状态容器。

职责：

- 管理消息流
- 管理处理状态
- 管理工具确认请求
- 管理模型选中与连接状态
- 发起 run 流式调用

当前关键设计：

- `checkConnection()` 已改为内部 `try/catch` 兜底
- `/api/agent/health` 失败不会再把页面炸掉
- `sendMessage()` 使用流式接口 `/api/agent/runs/stream`
- 确认与拒绝操作对应后端工具执行恢复链路

当前已知事实：

- `AgentTrigger` 每 30s 轮询一次 `checkConnection()`
- 该轮询当前是可接受的，因为后端 `agent/health` 已轻量化

## 3.5 `src/lib/apiConfig.ts`

这是 Provider 配置前端层。

职责：

- 拉取 provider 列表
- 创建/更新/删除 provider
- 切换 active provider
- 维护前端缓存

注意事项：

- 前端只拿到脱敏信息与 `hasApiKey`
- 不能依赖前端缓存作为真实 provider 权威源

## 4. 后端模块架构

## 4.1 App 装配层

入口文件：

- [server/src/index.ts](D:\project\Table\server\src\index.ts)
- [server/src/app.ts](D:\project\Table\server\src\app.ts)

`createApp()` 做的事：

1. 创建 Fastify 实例
2. 注册统一 `onRequest` 鉴权钩子
3. 注册全局错误处理器
4. 注册业务模块路由
5. 注册 projection runtime

当前注册顺序：

- `/api/health`
- `/api/auth`
- `/api/tasks`
- `/api/finance`
- `/api`
  - `knowledge`
  - `search`
  - `providers`
  - `agent`
- `/api/maintenance`

## 4.2 鉴权层：`server/src/shared/auth.ts`

这是当前后端最值得研发关注的基础设施之一。

职责：

- 解析请求用户上下文
- 校验用户身份
- 确保用户记录存在
- 按需补齐用户基线数据

### 当前实现演进

之前存在两个隐性问题：

1. 每个请求都重复跑重型基线补齐
2. 当任务和财务都为空时，会把默认样本数据偷偷补回去

当前已修复：

- 增加 `ensureUserBaselineOnce()`
- 对每个 `userId` 做进程内去重
- 基线样本数据只在“首次引导条件成立”时初始化
- `GET /api/agent/health` 通过 `ensureBaseline: false` 跳过重型基线补齐

### 当前补齐逻辑

补齐内容包括：

- `user_settings`
- `knowledge_base`
- 默认 provider（若配置了默认 provider 环境变量）
- 首次引导用任务/财务样本数据

设计含义：

- 普通业务请求不应承载“不断重建默认工作区”的责任
- 后续如引入更正式的 onboarding，应从这里拆出去

## 4.3 统一错误层：`server/src/shared/http.ts`

职责：

- 映射 Zod 校验错误
- 映射鉴权错误
- 返回统一结构的基础设施错误

当前返回语义：

- `400`：校验失败
- `401/403`：鉴权失败
- `503`：基础设施或运行错误

如果未来要做统一错误码治理，应从这里入手，而不是在各模块零散补。

## 6. 异步投影架构

## 6.1 为什么需要 outbox

如果任务/财务在写入时直接同步改知识库，会带来：

- 业务模块耦合
- 事务边界复杂
- 页面层和业务层混杂
- 失败恢复困难

因此当前采用 outbox。

## 6.2 Runtime 入口

- [server/src/modules/projection/runtime.ts](D:\project\Table\server\src\modules\projection\runtime.ts)

## 6.3 Runtime 时序

```text
业务请求写入 tasks / finance
  -> 同事务写入 projection_outbox_events
  -> runtime onReady 启动轮询
  -> list pending events
  -> claim event
  -> processKnowledgeProjectionEvent(event)
  -> processed / reschedule
```

## 6.4 运行机制

- `onReady` 启动轮询
- `kickProjectionRuntime()` 可触发快速消费
- 每个事件先 claim，再执行
- 失败后写回 attempts 与错误信息

这保证了：

- 事件不会被多个 worker 同时处理
- 短期失败可重试
- 知识投影最终一致

## 8. 关键时序说明

## 8.1 任务创建时序

```text
UI -> src/db/taskDB.add
   -> POST /api/tasks
   -> tasks route -> tasks service -> tasks repository
   -> 写 tasks
   -> 写 projection_outbox_events
   -> 返回 task DTO
   -> 前端刷新 task store
   -> 前端调度 knowledge refresh
   -> runtime 异步消费 outbox
   -> knowledge 投影更新
```

## 8.2 财务删除时序

```text
UI -> src/db/financeDB.delete
   -> DELETE /api/finance/:id
   -> finance service 软删除/删除记录
   -> 写 projection_outbox_events
   -> 前端更新 finance store
   -> runtime 异步消费 outbox
   -> knowledge 投影同步删除/更新
```

## 8.3 知识实体编辑时序

```text
UI -> src/kb/sync.ts upsertKnowledgeEntity
   -> PATCH /api/knowledge/entities/:id
   -> knowledge service -> 结构化表更新
   -> 前端 refreshKnowledgeDataset()
   -> GET /api/knowledge/dataset
   -> 页面重新渲染
```

## 8.4 智能体运行时序

```text
UI -> AgentContext.sendMessage
   -> POST /api/agent/runs/stream
   -> 创建 agent_run
   -> 读取 active provider
   -> 调用模型
   -> 写 agent_messages / tool_executions / snapshots
   -> SSE 推送 status / completed / waiting_confirmation
   -> 前端更新消息与确认状态
```

## 8.5 工具确认恢复时序

```text
UI 点击确认
   -> POST /api/agent/runs/:id/tools/:toolExecutionId/confirm
   -> 读取 tool_execution
   -> 读取最新 snapshot
   -> 恢复 pending tool 调用上下文
   -> 继续执行
   -> 写新消息、工具结果、状态
   -> 返回 run detail
   -> 前端刷新对话状态
```

## 9. 当前真实约束与注意事项

## 9.1 不要新增本地权威状态

禁止再把以下内容当权威源扩展：

- `localStorage`
- 本地 JSON 文件
- 页面内临时数组/对象

## 9.2 不要让业务模块互相直写

例如：

- `tasks` 不能直接写知识表
- `finance` 不能直接写知识表
- 页面不能同时改任务表和知识表

## 9.3 健康检查必须轻量

`GET /api/agent/health` 已经因为历史实现过重而触发过 504。

后续规则：

- health 接口只做轻量状态检查
- 禁止在 health 接口里做重型初始化或全量校验

## 9.4 请求封装必须区分有无 body

项目里已经出现过：

- 空 body 却带 `Content-Type: application/json`

后续规则：

- 只有真的有 body 时才设置 JSON content-type

## 9.5 用户基线补齐只能做“首次引导”

当前已经修复：

- 普通请求不会反复补齐默认样本数据

后续规则：

- onboarding 逻辑应显式设计
- 不能把默认演示数据恢复机制藏在鉴权钩子里

## 10. 研发扩展建议

## 10.1 如果新增业务模块

建议结构：

```text
server/src/modules/<domain>/
  routes.ts
  service.ts
  repository.ts
  dto.ts
  schema.ts
```

同时：

- 前端在 `src/db/` 或对应 `src/lib/` 增加 facade
- 如需进入知识域，走 outbox

## 10.2 如果新增智能体工具

建议修改点：

- `server/src/modules/agent/executor.ts`
- 工具元数据定义
- 前端消息/工具展示层

必须满足：

- 可记录
- 可确认
- 可恢复

## 10.3 如果新增知识能力

优先修改点：

- knowledge repository
- knowledge routes
- dataset assembly

不要做的事：

- 不要重新引入文件权威源
- 不要让页面直接操作 dataset 全量对象作为日常 CRUD

## 11. 当前研发结论

当前仓库已经不是“前端单体应用加一点后端补丁”，而是一个有明确边界的平台底座：

- 前端是 UI 和协调层
- 后端是业务和运行时层
- PostgreSQL 是权威数据层
- Outbox 是跨域异步桥梁
- 智能体已具备服务端运行骨架

从研发维护角度看，后续最重要的不是继续做兼容层，而是守住已经建立的这些边界。
