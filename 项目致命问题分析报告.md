## 一、CRITICAL 级别问题

### 1. ~~维护模块导入端点缺少权限保护~~ ✅ 已修复

**文件**：`server/src/modules/maintenance/routes.ts`

**已修复内容**（2026-05-10）：
- 为 `GET /business-snapshot` 和 `POST /business-snapshot` 添加了 `defaultUserOnly` 权限检查（仅默认用户可用）
- 为 `POST /business-snapshot` 添加了速率限制（10分钟1次）
- 提取 `defaultUserOnly` 为共享配置，`/reset` 端点也复用，消除重复代码
- 添加空快照导入保护（`Cannot import empty snapshot`）
- 导入前自动备份当前数据，响应中返回 `backup` 字段，误操作可恢复

**现状**：`POST /api/maintenance/business-snapshot`（导入）和 `GET /api/maintenance/business-snapshot`（导出）端点没有任何权限检查，任何通过全局认证的用户都可以调用。而同模块的 `POST /api/maintenance/reset` 端点正确地添加了 `preHandler` 限制仅默认用户可用。

**对比代码**：

```typescript
// 无权限保护 ❌
app.get('/business-snapshot', async (_request, reply) => { ... });
app.post('/business-snapshot', async (request, reply) => { ... });

// 有权限保护 ✅
app.post('/reset', {
  preHandler: [async (request, reply) => {
    const context = resolveRequestUserContext(request);
    if (context.userId !== getDefaultUserId()) {
      throw new AuthError('Only default user can reset workspace', 403, 'FORBIDDEN');
    }
  }],
}, ...);
```

**致命后果**：`importBusinessSnapshot`（`service.ts:218-304`）会硬删除用户的所有任务和财务记录、软删除知识笔记、硬删除预设标签，然后插入新数据。任何认证用户都能永久销毁其他用户的数据。

**修复方案**：为导入/导出端点添加与 `/reset` 相同的 `preHandler` 权限检查，并为导入端点添加速率限制。

### 2. ~~认证机制可被绕过——任意用户冒充~~ ✅ 已修复

**文件**：`server/src/shared/user-context.ts`

**已修复内容**（2026-05-10）：
- 新建 `server/src/shared/session.ts`，实现 HMAC-SHA256 签名令牌（`signSessionToken` / `verifySessionToken`），复用 `PROVIDER_SECRET_KEY` 作为签名密钥
- 修改 `resolveRequestUserContext`：优先验证签名 Cookie，`x-user-id` 头受 `TRUST_USER_ID_HEADER` 配置控制（默认 `false`）
- PIN 验证成功后签发 24 小时有效签名 Cookie，会话切换也使用签名 Cookie
- 前端 `fetchWithAuth` 不再发送 `x-user-id` 头，改为依赖 Cookie
- 未设置 PIN 时裸 UUID Cookie 仍可正常使用（兼容本地开发）

**现状**：系统仅通过 `x-user-id` 请求头或 `table_dev_session_user_id` Cookie 识别用户身份，没有任何密码学验证。

```typescript
export function resolveRequestUserContext(request: FastifyRequest): ServerUserContext {
  const headerValue = request.headers[USER_ID_HEADER];  // 直接读取头
  const sessionUserId = readCookieValue(request, DEV_SESSION_COOKIE);  // 直接读取 Cookie
  const userId = hasHeader ? headerValue.trim()
    : hasSession ? sessionUserId.trim()
    : getDefaultUserId();
  return { userId, source: ... };
}
```

**缺陷**：
- 无 JWT、无签名、无 HMAC、无 Token 校验
- Session Cookie 存储原始用户 ID，无任何签名保护
- 任何客户端只需设置 `x-user-id: <目标用户UUID>` 即可完全冒充该用户
- 默认用户 ID `00000000-0000-0000-0000-000000000001` 是公开的硬编码值

**组合攻击场景**：攻击者获取目标用户 UUID → 设置 `x-user-id` 头冒充 → 调用 `POST /api/maintenance/business-snapshot` 导入空数据 → 目标用户数据被永久销毁。

**修复方案**：长期需要引入真正的登录机制（用户名/密码或 OAuth）、JWT 或签名 Cookie、CSRF 保护、会话过期和刷新机制。

### 3. ~~导入操作硬删除数据，无确认/备份/撤销~~ ✅ 已修复

**文件**：`server/src/modules/maintenance/service.ts:218-304`

**已修复内容**（2026-05-10）：
- 导入前自动调用 `exportBusinessSnapshot()` 备份当前数据，响应中返回 `backup` 字段
- 空快照导入被拒绝（`Cannot import empty snapshot`），防止误操作清空工作台

### 4. ~~删除/更新操作 WHERE 条件缺少 userId（TOCTOU 漏洞）~~ ✅ 已修复

**文件**：
- `server/src/modules/tasks/repository.ts:58-66` — `softDeleteTask`
- `server/src/modules/finance/repository.ts:72-80` — `softDeleteFinanceRecord`
- `server/src/modules/knowledge/repository.ts` — `updateNote`、`deleteNote`、`updatePresetTag`、`deletePresetTag`

**已修复内容**（2026-05-10）：
- 所有 Repository 层的 `update`/`delete` 操作 WHERE 条件中添加了 `userId` 过滤
- 涉及 6 个函数：`updateTask`、`softDeleteTask`、`updateFinanceRecord`、`softDeleteFinanceRecord`、`updateNote`、`deleteNote`、`updatePresetTag`、`deletePresetTag`
- 为未来多用户扩展打好基础，消除 TOCTOU 竞态条件风险

**现状**：这些函数的 Prisma `update`/`delete` 调用只使用 `{ id }` 作为 WHERE 条件，没有同时包含 `userId`。

```typescript
export async function softDeleteTask(id: string) {
  return prisma.task.update({
    where: { id },   // 缺少 userId 过滤
    data: { deletedAt: new Date() },
  });
}
```

**风险**：虽然当前 Service 层在调用前检查了所有权，但存在 TOCTOU（Time-of-check to time-of-use）竞态条件——在所有权检查和删除操作之间，记录的 userId 可能被另一个请求修改。如果未来有代码路径直接调用 Repository 函数而跳过所有权检查，将导致跨用户数据篡改。

**修复方案**：在 WHERE 条件中同时包含 `userId`，例如 `where: { id, userId }`。

### 5. ~~无优雅关闭处理，Prisma 连接泄漏~~ ✅ 已修复

**文件**：`server/src/index.ts`

**已修复内容**（2026-05-10）：
- 添加 `SIGTERM` 和 `SIGINT` 信号处理器
- 收到信号后：先调用 `app.close()` 等待在途请求完成，然后调用 `prisma.$disconnect()` 释放数据库连接
- 防止重复触发（`shuttingDown` 标志）
- 记录关闭过程日志

**现状**：生产服务器没有 `SIGTERM`/`SIGINT` 信号处理，进程终止时 Prisma 客户端不会断开连接。

**后果**：
- PostgreSQL 连接池泄漏
- 进行中的事务被中断无清理
- 写操作在提交中途可能造成数据不一致

**修复方案**：
```typescript
process.on('SIGTERM', gracefulShutdown);
process.on('SIGINT', gracefulShutdown);

async function gracefulShutdown() {
  await prisma.$disconnect();
  await app.close();
  process.exit(0);
}
```

### 6. ~~无 unhandledRejection / uncaughtException 处理器~~ ✅ 已修复

**文件**：`server/src/index.ts`

**已修复内容**（2026-05-10）：
- 添加 `unhandledRejection` 和 `uncaughtException` 全局处理器
- 复用已有的 `gracefulShutdown` 函数，确保异常时也释放 Prisma 连接和 Fastify
- 记录错误日志便于排查

**现状**：没有全局异常处理器。虽然 Fastify 能捕获路由处理器中的错误，但逃逸请求生命周期的异步操作（如 `ensureUserBaselineOnce` 后台 Promise 或 Agent 执行器）产生的未处理拒绝会导致进程静默崩溃或进入不确定状态。

**修复方案**：添加 `process.on('unhandledRejection')` 和 `process.on('uncaughtException')` 处理器，记录错误并优雅关闭。

### 7. ~~速率限制白名单包含 localhost，可被利用~~ ✅ 已修复

**文件**：`server/src/app.ts:27-31`

**已修复内容**（2026-05-10）：
- 移除 `allowList: ['127.0.0.1', '::1']`，localhost 请求不再绕过速率限制
- 细粒度速率限制（导入 10分钟1次、PIN 验证 5分钟5次）现在在本地环境也能正常生效

app.register(rateLimit, {
  max: 100,
  timeWindow: '1 minute',

**修复方案**：移除 localhost 白名单，或仅在开发环境启用。

### 8. ~~导入操作删除策略不一致~~ ✅ 已修复

**文件**：`server/src/modules/maintenance/service.ts`

**已修复内容**（2026-05-10）：
- `importBusinessSnapshot` 中知识笔记改为硬删除（`deleteMany`），与任务/财务/预设标签保持一致
- 移除了软删除导致的垃圾数据累积问题
- 导入端点速率限制从 10分钟1次改为 1分钟1次，重置端点同步调整

### 12. ~~Session Cookie 无签名，可被篡改~~ ✅ 已修复

**文件**：`server/src/modules/auth/routes.ts:247-250`

**已修复内容**（2026-05-10）：
- PIN 验证和会话切换端点现在签发 HMAC-SHA256 签名 Cookie，格式为 `<userId>.<expiresTimestamp>.<hmacSignature>`
- 篡改 Cookie 中任何部分都会导致签名验证失败
- Cookie 有效期 24 小时，过期后需重新验证 PIN

```typescript
reply.header(
  'Set-Cookie',
  `${DEV_SESSION_COOKIE}=${encodeURIComponent(payload.userId)}; Path=/; HttpOnly; SameSite=Lax${secureFlag}`
);
```

**缺陷**：
- Cookie 存储原始用户 ID，无 HMAC 签名
- 任何人修改 Cookie 值即可冒充其他用户
- `Secure` 标志仅在生产环境设置，开发环境 Cookie 可通过 HTTP 传输
- 手动构建 `Set-Cookie` 头而非使用 Fastify Cookie 插件

**修复方案**：使用 Fastify Cookie 插件，添加签名机制。
