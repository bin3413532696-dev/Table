# Table 技术架构文档

## 1. 文档目的

本文档用于说明 `Table` 项目的当前技术架构、模块职责、数据流转方式、核心设计决策与扩展边界。

本文档基于当前仓库中的真实代码实现编写，不是规划稿，也不是迁移方案稿。其目标是让研发、交付、维护人员能够快速回答以下问题：

- 系统由哪些核心层组成
- 前后端各自承担什么职责
- 数据的权威源在哪里
- 任务、财务、知识库、智能体、Provider 如何协同
- 当前系统如何扩展而不破坏已建立的职责边界

---

## 2. 架构定位

`Table` 已不再是“前端本地存储驱动的单机工作台”，当前定位是：

**可持续扩展的智能工作台平台底座**

其核心特征为：

- 前端负责交互、展示、局部缓存与页面编排
- 后端负责权威写入、业务规则、运行时编排与对外 API
- PostgreSQL 是业务数据的长期权威存储
- 知识库不再依赖本地文件或浏览器本地存储作为主数据源
- 智能体运行链路已后端化，前端只负责发起、展示、确认与恢复
- 任务与财务到知识库的同步通过异步投影完成，而不是页面层双写

---

## 3. 总体架构

### 3.1 分层结构

系统当前采用四层结构：

1. 前端应用层
2. 后端业务服务层
3. 数据持久化层
4. 异步投影与运行时基础设施层

### 3.2 逻辑视图

```text
+---------------------------+
| React Frontend            |
| 页面 / Context / Store    |
| DB Facade / Sync Engine   |
+------------+--------------+
             |
             | HTTP / JSON
             v
+---------------------------+
| Fastify API Server        |
| Auth / Tasks / Finance    |
| Knowledge / Search        |
| Providers / Agent         |
| Maintenance               |
+------------+--------------+
             |
             | Prisma
             v
+---------------------------+
| PostgreSQL                |
| Users / Tasks / Finance   |
| Knowledge Structured Data |
| Providers / Agent Runs    |
| Projection Outbox         |
+------------+--------------+
             |
             | Poll + Claim + Process
             v
+---------------------------+
| Projection Runtime        |
| Outbox -> Knowledge View  |
+---------------------------+
```

## 5. 目录与模块职责

### 5.1 根目录

- `src/`
  - 前端应用
- `server/`
  - 后端服务
- `prisma/`
  - 数据模型与迁移
- `scripts/`
  - 验证与辅助脚本
- `README.md`
  - 交付入口说明

### 5.2 前端模块

#### `src/db/`

前端业务数据访问门面。

职责：

- 封装 `tasks` / `finance` / `maintenance` 相关 API
- 将服务端返回结果同步到前端 store
- 触发知识库投影刷新

#### `src/sync/`

统一同步引擎。

职责：

- 管理知识库同步防抖
- 统一知识库权威数据读写端点
- 管理同步状态、错误与重试计数

当前知识库权威接口：

- `GET /api/knowledge/dataset`
- `PUT /api/knowledge/dataset`
- `GET /api/knowledge/metadata`

#### `src/kb/`

知识库前端访问与组装层。

职责：

- 维护页面所需知识数据集结构
- 封装知识实体、文档、断言、Ontology 的远程 CRUD
- 对接统一同步引擎

#### `src/agent/`

智能体前端状态层。

职责：

- 管理对话消息状态
- 发起智能体运行请求
- 处理工具确认与拒绝
- 轮询或读取运行时状态

#### `src/lib/`

共享能力层。

职责：

- 鉴权请求封装
- Provider 配置访问
- 搜索 API
- 智能体 API

#### `src/store/`

前端局部缓存层。

当前语义：

- 仅作为页面级缓存与交互态容器
- 不再作为任务、财务、知识库的长期权威源

### 5.3 后端模块

#### `server/src/modules/auth/`

职责：

- 请求用户上下文解析
- 默认用户/开发态用户体系支持
- 用户资料与会话切换

#### `server/src/modules/tasks/`

职责：

- 任务 CRUD
- 权威写入 PostgreSQL
- 触发知识投影事件

#### `server/src/modules/finance/`

职责：

- 财务记录 CRUD
- 权威写入 PostgreSQL
- 触发知识投影事件

#### `server/src/modules/knowledge/`

职责：

- 知识实体、文档、断言、关系、Ontology 的服务端权威读写
- 从结构化表组装知识数据集视图
- 导入、重置、全量替换场景支持

#### `server/src/modules/search/`

职责：

- 对知识结构化数据提供统一搜索能力

#### `server/src/modules/providers/`

职责：

- 用户级 Provider 配置管理
- 激活配置切换
- 密钥加密存储与脱敏返回

#### `server/src/modules/agent/`

职责：

- 智能体运行记录
- 消息与工具执行记录
- 多轮运行、工具执行与确认恢复
- 运行时健康检查

#### `server/src/modules/projection/`

职责：

- 管理 outbox 事件
- 轮询并消费待处理投影任务
- 将任务/财务业务数据投影到知识结构

#### `server/src/modules/maintenance/`

职责：

- 业务快照导入导出
- 工作台重置
- 开发与恢复工具

## 6. 关键设计原则

### 6.1 权威源唯一化

当前系统中，长期权威源应只有以下几类：

- PostgreSQL 业务表
- 由后端结构化表组装出的知识数据集视图

前端内存、前端 store、浏览器 `localStorage` 都不再承担主权威源角色。

### 6.2 读写职责分离

- 前端负责发起业务意图
- 后端负责执行校验、持久化与领域规则
- 前端不再直接维护跨模块双写逻辑

### 6.3 异步投影替代同步双写

任务与财务不再直接同步写入知识库，而是：

1. 业务写入各自表
2. 写入 projection outbox
3. 异步运行时消费事件
4. 更新知识结构化视图

这样可以降低耦合度，避免页面层维护多套一致性逻辑。

### 6.4 服务端化智能体运行

智能体不再依赖前端本地状态机直接执行。

当前模式为：

- 前端发起 run
- 后端创建 `agent_runs`
- 后端写入 `agent_messages`
- 后端记录 `tool_executions`
- 如遇确认型工具，则写入待确认快照
- 前端仅负责展示、确认与恢复

## 7. 数据架构

### 7.1 用户与配置

核心表：

- `users`
- `user_settings`
- `api_providers`

说明：

- 用户是整个系统的数据隔离边界
- Provider 是按用户维度配置的
- 用户设置承载主题、资料、通知、智能体偏好等

### 7.2 业务域数据

核心表：

- `tasks`
- `finance_records`

说明：

- 两者都是面向工作台核心业务的权威表
- 通过 `deleted_at` 支持软删除
- 通过 `version` 支持乐观版本演进

### 7.3 知识库数据

当前知识库采用“结构化拆表 + 组装视图”模式。

核心表：

- `knowledge_bases`
- `knowledge_entities`
- `knowledge_documents`
- `knowledge_assertions`
- `knowledge_relations`
- `knowledge_document_entities`
- `knowledge_assertion_evidence`
- `knowledge_ontology_classes`
- `knowledge_ontology_relations`

说明：

- `knowledge_bases` 当前更接近 metadata 容器与顶层聚合锚点
- 常规知识写入已不依赖单个大 JSON 作为真实权威源
- 对外的数据集接口返回的是结构化表组装后的视图

### 7.4 智能体运行数据

核心表：

- `agent_runs`
- `agent_messages`
- `tool_executions`
- `agent_run_state_snapshots`

说明：

- `agent_runs` 表示一次智能体运行
- `agent_messages` 保存对话消息序列
- `tool_executions` 保存工具调用与确认状态
- `agent_run_state_snapshots` 保存可恢复快照

### 7.5 异步投影数据

核心表：

- `projection_outbox_events`

说明：

- 用于承接任务/财务业务事件
- 后台运行时轮询、claim、执行、重试
- 是任务/财务与知识库之间的异步解耦层

## 8. 核心运行链路

### 8.1 任务写入链路

```text
Frontend taskDB
  -> POST/PATCH/DELETE /api/tasks
  -> Fastify tasks module
  -> Prisma 写入 tasks
  -> 写入 projection_outbox_events
  -> Projection runtime 消费
  -> 更新 knowledge 结构化数据
  -> 前端通过知识数据集接口刷新视图
```

### 8.2 财务写入链路

```text
Frontend financeDB
  -> /api/finance
  -> Fastify finance module
  -> Prisma 写入 finance_records
  -> 写入 projection_outbox_events
  -> Projection runtime 消费
  -> 更新 knowledge 结构化数据
```

### 8.3 知识库写入链路

```text
Frontend kb sync/api facade
  -> /api/knowledge/*
  -> knowledge module
  -> Prisma 写入知识结构化表
  -> /api/knowledge/dataset 组装视图返回
```

### 8.4 智能体运行链路

```text
Frontend AgentContext
  -> /api/agent/runs 或 /api/agent/runs/stream
  -> agent module 创建 run
  -> 读取激活 Provider
  -> 调用大模型接口
  -> 写入消息、工具执行记录、快照
  -> 前端展示运行结果/确认请求
```

### 8.5 Provider 配置链路

```text
Settings Page
  -> /api/providers
  -> providers module
  -> api_providers
  -> 密钥加密存储
  -> 返回脱敏配置
```

## 9. 前端架构细节

### 9.1 页面层

页面位于 `src/pages/`，主要承担：

- 路由入口
- 页面交互状态
- 与数据门面的组合

页面不应承担跨领域同步规则与底层持久化逻辑。

### 9.2 数据访问门面

前端通过 `src/db/index.ts` 对任务、财务、维护能力进行统一访问。

这样做的收益：

- 页面不直接关心 HTTP 细节
- 可以在一个地方做错误映射
- 可以在一个地方做本地缓存刷新

### 9.3 同步引擎

`src/sync/SyncEngine.ts` 是当前知识同步的唯一协调点。

它负责：

- 防抖
- 同步状态
- 服务端知识数据读取
- 全量知识数据写入

当前不再使用 websocket 作为知识刷新主机制，而是：

- 元数据轮询
- 显式刷新
- 写后同步

### 9.4 智能体前端状态机

`src/agent/AgentContext.tsx` 承担对话状态管理。

当前设计特点：

- 统一错误兜底
- 统一连接检查
- 手动终止支持
- 确认型工具流程支持
- 不再让运行时健康检查失败直接炸页面

## 10. 后端架构细节

### 10.1 App 装配

后端入口位于：

- `server/src/index.ts`
- `server/src/app.ts`

`createApp()` 中统一注册模块与钩子：

- `auth`
- `health`
- `tasks`
- `finance`
- `knowledge`
- `search`
- `providers`
- `agent`
- `maintenance`
- projection runtime

### 10.2 鉴权与用户上下文

每个请求会进入统一认证钩子。

当前特点：

- 默认开发态支持本地默认用户
- 统一解析 `x-user-id` / session
- 用户基线补齐已做按用户去重
- `GET /api/agent/health` 已跳过重型基线补齐

### 10.3 错误处理

后端通过统一错误处理器返回标准化错误响应。

当前分层：

- 参数校验错误
- 鉴权错误
- 基础设施错误

### 10.4 Projection Runtime

`server/src/modules/projection/runtime.ts` 负责异步消费投影事件。

当前机制：

- `onReady` 时启动轮询
- 读取待处理 outbox 事件
- claim 事件
- 处理知识投影
- 成功标记 processed
- 失败记录错误并重试

这是当前任务/财务与知识库解耦的关键。

## 12. 安全与配置

### 12.1 Provider 密钥

Provider 密钥不会以明文长期回传给前端。

当前策略：

- 服务端加密存储
- 前端读取时只拿到脱敏预览与 `hasApiKey`
- 真正运行时由后端解密后使用

### 12.2 用户隔离

多数核心表都带 `user_id`，当前系统以用户作为数据边界。

### 12.3 当前限制

当前仍存在的现实限制：

- 仍以开发态默认用户为主
- 还没有正式租户体系
- 还没有正式 RBAC 权限模型
- 还没有完整审计日志体系

这些属于未来平台化扩展方向，不影响当前工作台底座交付。

## 14. 扩展建议

### 14.1 新业务模块

新增业务模块时应遵循：

1. 新权威写入逻辑进入 `server/src/modules/*`
2. 数据结构经 `prisma/schema.prisma` 演进
3. 前端通过 facade 或 API 层访问
4. 如需进入知识域，优先走 outbox + projection

### 14.2 新智能体能力

新增智能体能力时应遵循：

1. 前端只做展示与交互
2. 工具定义、执行与恢复放在后端
3. 所有运行状态可回放、可确认、可审计

### 14.3 新搜索或知识能力

新增知识能力时应遵循：

1. 优先在知识结构化表上扩展
2. 不回退到文件或本地存储权威模式
3. 对外继续通过组装视图暴露

## 15. 当前架构结论

当前 `Table` 的真实架构已经完成从单机前端应用到平台底座的关键跨越：

- 权威源从浏览器本地迁移到服务端与数据库
- 跨领域同步从页面层双写迁移到异步投影
- 智能体从前端执行迁移到后端运行与记录
- Provider 从本地配置迁移到后端权威管理
- 知识库从本地快照迁移到结构化数据与组装视图

因此，当前系统已经具备作为“智能工作台平台底座”继续扩展的基础条件。

后续演进重点不再是“去本地化”止血，而应转向：

- 权限与用户体系完善
- 搜索与知识能力增强
- 智能体运行时能力增强
- 性能与观测性建设
