## 1. 数据层缺陷

### 1.1 `createUseDB` 钩子定义后未被任何组件使用

**文件**: `src/db/index.ts:846`  
**严重程度**: 高

`createUseDB` 是一个工厂函数，返回响应式 `useDB` 钩子，支持订阅数据变更后自动重新获取。但该函数在全项目中未被任何组件引用。

```typescript
export function createUseDB(React: typeof import('react')) {
  return function useDB<T>(fetcher, dependencies): T | null { ... };
}
```

所有页面（Dashboard、Notes、Tasks、Finance）依然使用 `useEffect` + `useState` 手动加载数据，导致：
- 重复的数据加载模板代码（每个页面写一遍 `useEffect` → `getAll()` → `setState`）
- 一个页面的数据变更不会自动通知其他已挂载的页面（如修改任务后切换到 Dashboard，任务计数不会刷新）
- 新的 `subscribe` 机制仅有定义无消费者

### 1.2 `migrateLinksFromTitleToId()` 在每次 getAll/getByFolder 时执行

**文件**: `src/db/index.ts:492,549,554`  
**严重程度**: 高

迁移函数在每次 `noteDB.getAll()` 和 `noteDB.getByFolder()` 调用时都会执行。它遍历所有笔记检查是否需要迁移。迁移是一次性操作，数据稳定后不应重复执行。

```typescript
async getAll(): Promise<Note[]> {
  await migrateLinksFromTitleToId(); // 每次调用都执行
  return noteStore.getAll().sort(...);
}
```

该方法内部每次遍历所有笔记，检查链接是标题格式还是 ID 格式。数据量增大时，每个列表渲染/搜索请求都会产生 O(n) 开销。

### 1.3 `noteDB.updateLinks()` 和 `noteDB.delete()` 替换全部记录

**文件**: `src/db/index.ts:655-671, 629-636`  
**严重程度**: 高

更新链接和删除笔记时，使用 `noteStore.replaceAll(updatedRecords)` 替换整个笔记集合，而非仅更新受影响的记录。这在笔记数量大时造成不必要的数据拷贝和写入。

```typescript
// delete() - 更新所有笔记的 links/backlinks，再过滤掉被删笔记
const updatedRecords = noteRecords.map(n => { ... }).filter(r => r.id !== id);
noteStore.replaceAll(updatedRecords);

// updateLinks() - 更新所有笔记的 links/backlinks
const updatedRecords = noteRecords.map(n => { ... });
noteStore.replaceAll(updatedRecords);
```

### 1.4 `initDB()` 已成为空操作

**文件**: `src/db/index.ts:421`  
**严重程度**: 中

```typescript
export const initDB = async () => {
  return true;
};
```

该函数现在只是 `async () => true`，不做任何实际工作。但 Finance 页面仍然在 `useEffect` 中调用 `initDB()`（`src/pages/Finance/index.tsx:30`），这是无意义的调用，容易让后续维护者困惑。

### 1.5 `getStats()` 中 JSON.stringify 全量数据计算大小

**文件**: `src/db/index.ts:812-819`  
**严重程度**: 低

```typescript
getStats() {
  const financeData = { finance: financeRecords, tasks: taskRecords, notes: noteRecords, folders: folderRecords };
  return {
    ...
    totalSize: JSON.stringify(financeData).length // 全量序列化仅为了计算大小
  };
}
```

每次调用 `getStats()` 都会对全部数据进行 JSON 序列化。对于包含大量笔记的数据集，这是不必要的性能浪费。可改用增量方法或在数据变更时缓存大小。

### 1.6 `scheduleSync()` 使用 `setTimeout` 而非 `requestAnimationFrame` 或空闲回调

**文件**: `src/db/index.ts:228-236`  
**严重程度**: 低

数据同步使用 `setTimeout(..., 1500)` 防抖。在组件卸载或页面关闭时不会取消同步任务，可能导致页面关闭后发起请求。使用 `requestIdleCallback` 或 `Navigator.sendBeacon` 在页面关闭场景下更可靠。

### 1.7 `noteDB.search()` 搜索算法基础

**文件**: `src/db/index.ts:639-645`  
**严重程度**: 中

```typescript
async search(query: string): Promise<Note[]> {
  const lowerQuery = query.toLowerCase();
  return noteStore.filter(n =>
    n.title.toLowerCase().includes(lowerQuery) ||
    n.content.toLowerCase().includes(lowerQuery)
  );
}
```

仅做大小写不敏感的 `includes` 匹配，无模糊搜索、无相关性排序、无高亮。项目中已安装 `fuse.js`，但 Notes 页面和搜索命令面板使用不同的搜索机制（命令面板用 Fuse.js，笔记搜索用此方法），二者结果不一致。

### 1.8 `folderDB.getTree()` 仅支持单层嵌套

**文件**: `src/db/index.ts:743-749`  
**严重程度**: 中

```typescript
async getTree(): Promise<Array<Folder & { children: Folder[] }>> {
  const rootFolders = folderStore.filter(f => f.parentId === null);
  return rootFolders.map(folder => ({
    ...folder,
    children: folderStore.filter(f => f.parentId === folder.id)
  }));
}
```

仅支持根文件夹 → 子文件夹两层。`children` 中的文件夹若有自己的 `parentId` 指向父文件夹，其自身的子文件夹不会被递归获取。FileTree 组件虽然支持递归渲染，但数据层限制了嵌套深度。

## 2. 组件架构缺陷

### 2.1 FileTree 接收过多 Props，与 Notes 页面高度耦合

**文件**: `src/pages/Notes/components/FileTree.tsx`  
**严重程度**: 高

FileTree 接收了大量 props（从 Notes 页面传递），粗略统计约 15-20 个：

| Prop | 用途 |
|---|---|
| folders, notes, currentNote, setCurrentNote | 数据源与选中状态 |
| selectedFolder, setSelectedFolder, onFolderSelect | 文件夹筛选 |
| onNoteCreate, onNoteDelete, onFolderCreate, onFolderDelete, onFolderRename | CRUD 回调 |
| searchResults, isSearching | 搜索状态 |
| expandedFolders, setExpandedFolders, onToggleExpand | 展开/折叠控制 |
| onDragStart, onDragOver, onDrop | 拖拽回调 |
| contextMenu, setContextMenu | 右键菜单 |

过多的 props 表明 FileTree 承担了过多职责（文件树显示、拖拽、右键菜单、搜索过滤、展开折叠），且与父组件的状态管理紧密耦合。将部分状态提升到 FileTree 内部或将数据操作封装为自定义 hooks 可降低耦合度。

### 2.2 Notes 页面承担过多职责

**文件**: `src/pages/Notes/index.tsx`  
**严重程度**: 高

Notes 页面同时管理：
- 笔记数据的加载与写入
- 三栏布局的拖拽与折叠状态
- 搜索模式与过滤逻辑
- 命令面板的开关
- 当前笔记/文件夹/标签的选择状态
- 右侧面板的 Tab 切换

代码行数超过 450 行，加上 6 个子组件后整个笔记模块超过 2000 行。缺少将数据操作、布局状态、搜索逻辑分离到各自 hooks/context 中的重构。

### 2.3 `useTheme` 存根文件仍存在于 hooks 目录

**文件**: `src/hooks/useTheme.ts` 与 `src/contexts/ThemeContext.tsx`  
**严重程度**: 中

两个文件导出了同名 `useTheme`，但实现不同：
- `src/hooks/useTheme.ts`: 恒返回 `{ isDark: false }` 的存根
- `src/contexts/ThemeContext.tsx`: 完整的 light/dark 切换上下文

存根文件未被任何组件引用，但仍存在于项目中。新开发者可能导入错误的 `useTheme`，获得与预期不符的行为。

### 2.4 设置页使用 `useTheme` 来自 ThemeContext 但安全选项卡中引入了它

**文件**: `src/pages/Settings/index.tsx:5,151`  
**严重程度**: 低

从系统提示的 diff 可以看到，`SecuritySettings` 组件引入了 `useTheme`：

```typescript
import { useTheme } from '../../contexts/ThemeContext';
// ... 在 SecuritySettings 中：
const { theme, toggleTheme } = useTheme();
```

但 `theme` 和 `toggleTheme` 在 *SecuritySettings* 中未被使用。主题切换功能被导入到安全选项卡中，应属于独立的选项卡或放置在更合适的位置。

### 2.5 `createUseDB` 使用 `React.useCallback` 但 `fetcher` 引用不稳定

**文件**: `src/db/index.ts:854-858`  
**严重程度**: 中

```typescript
const fetchData = React.useCallback(async () => {
  setLoading(true);
  const result = await fetcher();
  setData(result);
  setLoading(false);
}, [fetcher]); // fetcher 作为依赖
```

如果调用者每次渲染创建新的 `fetcher` 函数（如箭头函数），`useCallback` 将不断重建，导致无限循环或重复请求。`fetcher` 应在调用侧稳定引用，但文档/类型中未说明此要求。

### 2.6 GraphView 中节点选中回调引用不稳定

**文件**: `src/pages/Notes/components/GraphView.tsx`  
**严重程度**: 中

`onSelectNote` 回调从父组件传入，若父组件每次渲染传递新箭头函数，D3 仿真会检测到 `props.onSelectNote` 引用变化而重建。

```typescript
// 在 Notes/index.tsx 中
<GraphView
  onSelectNote={(note) => setCurrentNote(note)} // 每次渲染新函数
/>
```

## 3. 主题与样式缺陷

### 3.1 组件大量使用硬编码颜色值

**全项目**  
**严重程度**: 高

虽然已定义 CSS 变量（`--bg-primary`, `--text-primary` 等）并有 `ThemeContext` 支持 dark 模式切换，但几乎所有组件仍使用硬编码的 Tailwind 颜色类：

| 文件 | 问题代码 |
|---|---|
| Dashboard | `bg-white` `text-gray-900` `border-gray-200` |
| Tasks | `bg-white` `text-gray-900` `border-gray-200` |
| Finance | `bg-white` `text-gray-900` `border-gray-200` |
| Settings | `bg-white` `text-gray-900` `border-slate-200` |
| Notes/Editor | `bg-white` `border-gray-200` |
| FileTree | `bg-white` `hover:bg-gray-50` |

这意味着切换到 dark 模式时，大量组件将保持浅色背景和深色文本，出现视觉混乱。正确的做法应是使用 CSS 变量（`bg-[var(--bg-primary)]`）或 Tailwind 的 `dark:` 变体。

### 3.2 CSS 变量主题体系仅定义了背景和文本色

**文件**: `src/styles/index.css`  
**严重程度**: 中

定义的 CSS 变量：
```
--bg-primary, --bg-secondary, --bg-tertiary, --bg-card
--text-primary, --text-secondary, --text-muted
--border-primary, --border-secondary
```

缺少组件级语义化变量（如 `--color-primary-500`, `--color-danger-500`, `--sidebar-bg`, `--editor-bg`）。当前组件硬编码 `bg-blue-500`、`text-emerald-500` 等 Tailwind 语义色，dark 模式下这些颜色无法自动适配，需要逐一定义 `dark:` 变体。

### 3.3 缺少 `prefers-color-scheme` 媒体查询兜底

**文件**: `src/contexts/ThemeContext.tsx`、`index.html`  
**严重程度**: 低

`ThemeContext` 初始化时读取 localStorage，若无存储值则使用 `prefers-color-scheme`。但在 `index.html` 的内联脚本中，直接应用了 `'light'` 主题，未检查用户系统偏好，导致首次访问时即使系统为暗色模式也会闪现浅色主题。

### 3.4 滚动条样式仅使用了 Webkit 前缀

**文件**: `src/styles/index.css:45-59`  
**严重程度**: 低

`.scrollbar-thin` 使用 `::-webkit-scrollbar` 相关伪类，在 Firefox 等非 Webkit 浏览器中无效。Firefox 使用 `scrollbar-width: thin` 和 `scrollbar-color` 属性，未做兼容。

## 4. 编辑器缺陷

### 4.1 `insertText` 依赖 DOM 选区而非 CodeMirror API

**文件**: `src/pages/Notes/components/Editor.tsx`  
**严重程度**: 高

格式化工具栏（加粗、斜体、标题等）的 `insertText` 函数使用 DOM 原生 `window.getSelection()` 和 `document.querySelector('.cm-content')` 操作选区，而非 CodeMirror 的 `view.dispatch` API。

```typescript
// 问题模式
const selection = window.getSelection();
const selectedText = selection?.toString();
const contentEl = document.querySelector('.cm-content');
const content = contentEl?.textContent || '';
const start = content.indexOf(selectedText); // 只匹配首次出现
```

问题：
- `content.indexOf(selectedText)` 若选中文本多次出现，仅替换第一处
- CodeMirror 的虚拟 DOM 可能未同步到真实 DOM，选区信息不准
- Vim 模式下插入（Insert）和普通（Normal）模式状态未考虑

### 4.2 编辑器全屏模式下工具栏不跟随

**文件**: `src/pages/Notes/components/Editor.tsx`  
**严重程度**: 低

编辑器全屏时，格式化工具栏不固定在顶部，而是随页面滚动。在长文档编辑中，用户需要滚动到顶部才能访问格式化按钮，降低了全屏模式的实际可用性。

### 4.3 Split 模式下预览无防抖

**文件**: `src/pages/Notes/components/Editor.tsx`  
**严重程度**: 中

在分屏模式（编辑 + 预览并排）下，`react-markdown` 预览组件在每次内容变更时重新渲染。对于长文档，每次键盘输入都会触发完整的 Markdown 解析和渲染树重建。建议使用防抖（300ms）延迟预览更新。

### 4.4 编辑器无自动保存指示

**文件**: `src/pages/Notes/components/Editor.tsx`  
**严重程度**: 中

编辑器内容变化时调用 `handleUpdateNote` 写入 localStorage，但用户界面上无保存状态指示（"已保存"、"正在保存..."、"未保存的更改"）。用户无法感知数据是否已持久化。

## 5. 性能缺陷

### 5.1 Notes 页每次更新重新加载全量笔记

**文件**: `src/pages/Notes/index.tsx`  
**严重程度**: 高

`loadData()` 在每次笔记编辑 → `handleUpdateNote` → `noteDB.getAll()` + `folderDB.getAll()` 的链路中重新加载全部笔记和文件夹数据。即使只修改了一个字符，也重新读取整个数据集。

```typescript
const loadData = async () => {
  const [notes, folders] = await Promise.all([
    noteDB.getAll(),
    folderDB.getAll(),
  ]);
  setAllNotes(notes);
  setAllFolders(folders);
};
```

未被 `createUseDB` 或数据订阅机制优化。

### 5.2 TagCloud 每次渲染重新计算全部标签统计

**文件**: `src/pages/Notes/components/TagCloud.tsx`  
**严重程度**: 中

```typescript
const tagCounts = notes.reduce(...) // 每次渲染遍历所有笔记
const maxCount = Math.max(...Object.values(tagCounts));
```

每次渲染时遍历全部笔记计算标签频率，未使用 `useMemo` 缓存。若笔记达数百条且触发频繁渲染（如编辑器内容变化），会造成不必要的计算。

### 5.3 Dashboard 重复计算统计数据

**文件**: `src/pages/Dashboard/index.tsx`  
**严重程度**: 中

Dashboard 在每个渲染周期中多次调用 `toFixed(1)`、`Date` 解析等操作，缺少 `useMemo` 缓存。尤其 `taskDB.getStats()` 和 `financeDB.getStats()` 在无数据变更时不应重复调用。

### 5.4 右键菜单渲染开销

**文件**: `src/pages/Notes/components/FileTree.tsx`  
**严重程度**: 低

右键菜单（ContextMenu）使用 Framer Motion 的 `AnimatePresence`，每个文件/文件夹节点都可能触发菜单的挂载/卸载动画。在频繁的右键点击操作中，动画组件的创建和销毁会造成轻微卡顿。

## 6. 安全与可靠性缺陷

### 6.1 无数据恢复/回退机制

**严重程度**: 高

当 localStorage 数据损坏（如 `JSON.parse` 失败）时：
- `loadFromStorage` 静默返回空数组 `[]`
- 用户的数据被静默丢弃而无任何提示
- 无备份恢复建议或自动从 data/ 目录恢复的逻辑

```typescript
function loadFromStorage<T>(key: string): T[] {
  try {
    const data = localStorage.getItem(key);
    return data ? JSON.parse(data) : []; // 解析失败返回空数组
  } catch {
    return []; // 静默吞掉错误
  }
}
```

### 6.2 无 Content Security Policy

**文件**: `index.html`  
**严重程度**: 中

`index.html` 中没有 `Content-Security-Policy` meta 标签。项目引入了外部 CDN 资源（font-awesome），且使用 `innerHTML` 或 `dangerouslySetInnerHTML` 渲染 Markdown 内容，增加了 XSS 攻击面。

### 6.3 外部 CDN 依赖 font-awesome

**文件**: `index.html:8`  
**严重程度**: 中

```html
<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css">
```

项目使用 `lucide-react` 作为图标库，因此 font-awesome 的引入可能是冗余的。CDN 依赖带来：
- 额外的网络请求延迟
- CDN 不可用时的样式丢失（无 fallback）
- 隐私问题（CDN 可追踪访问）
- 安全风险（CDN 被篡改时引入恶意代码）

### 6.4 PIN 码无最大尝试次数硬限制

**文件**: `src/components/PinLock.tsx`  
**严重程度**: 中

虽然 PinLock 组件实现了 3 次尝试后 30 秒冷却，但此限制通过前端 JavaScript 实现。用户可刷新页面重置尝试计数，或直接修改 localStorage 移除 PIN。真正的安全保护应结合后端或 Service Worker 实现。

### 6.5 无 Service Worker / PWA 支持

**严重程度**: 中

所有数据存储在浏览器 localStorage，但无 Service Worker 注册。意味着：
- 浏览器清除"站点数据"（包括 localStorage）会导致全部数据丢失
- 无离线访问能力（虽然数据在本地，但 React 应用仍需网络加载）
- 无安装为 PWA 的能力

## 7. 代码质量问题

### 7.1 跨模块数据操作散布在多个组件中

**严重程度**: 高

数据操作逻辑（如笔记移动时更新文件夹的 `noteCount`）被分散在多个位置：
- `noteDB.add()` 中更新文件夹 `noteCount`
- `noteDB.update()` 中当 `folderId` 变更时调整旧/新文件夹的 `noteCount`
- `noteDB.delete()` 中减少文件夹 `noteCount`
- `folderDB.delete()` 中递归处理子文件夹和笔记

这种分散式维护在涉及 3+ 个集合的操作中容易遗漏状态同步。

### 7.2 缺少 `useMemo` 和 `useCallback` 优化

**全项目**  
**严重程度**: 高

大量计算密集型操作未使用 `useMemo` 或 `useCallback`：
- TagCloud 的标签频率计算
- Finance 的模型统计数据聚合
- Dashboard 的统计摘要计算
- FileTree 的排序和过滤结果
- 事件处理函数（FileTree 的 `onDrop`、`onDragOver` 等）每次渲染重建

导致子组件（尤其是 `React.memo` 未包裹的组件）不必要的重渲染。

### 7.3 CommandPalette 搜索结果无虚拟化

**文件**: `src/pages/Notes/components/CommandPalette.tsx`  
**严重程度**: 中

```typescript
{filteredResults.map((item, index) => (
  <div key={item.id || item.type}>
    ...
  </div>
))}
```

所有搜索结果一次性渲染到 DOM，无虚拟化。笔记和文件夹数量大时搜索性能下降。

### 7.4 Tool 页面无键盘快捷键

**文件**: `src/pages/Tools/index.tsx`  
**严重程度**: 低

与 Notes 页面的 Ctrl+K 命令面板不同，Tools 页面无键盘快捷键支持。计算器操作只能通过点击屏幕按钮完成，不能通过键盘数字键输入。

### 7.5 Finance 页面存在 `useStatePartial` 泛型语法问题

**文件**: `src/pages/Finance/index.tsx:32`  
**严重程度**: 低

根据初步扫描结果，Finance 页面可能存在：


### 7.6 `noteDB.getAll()` 每次调用重新排序

## 8. 构建与部署缺陷

### 8.1 生产构建未做代码分割

**文件**: `webpack.config.js`  
**严重程度**: 高

整个应用打包为单个 `bundle.js`，包含所有页面和依赖（CodeMirror、D3、Recharts、Framer Motion 等）。首次加载需要下载全部代码，即使只访问仪表盘页面。路由级懒加载可显著减小初始包体积。

| 依赖 | 估算大小 | 页面前置 |
|---|---|---|
| @uiw/react-codemirror + 扩展 | ~600KB | Notes |
| d3 | ~250KB | Notes (GraphView) |
| recharts | ~150KB | Finance |
| framer-motion | ~150KB | 全部页面 |
| react-markdown + remark-gfm | ~100KB | Notes |

### 8.2 无 CI/CD 配置

**严重程度**: 中

项目中无 GitHub Actions 或其他 CI 配置文件。每次提交后无法自动运行类型检查、构建验证或测试。代码质量保障依赖开发者本地手动执行 `npm run typecheck`。

### 8.3 `webpack.config.js` 中 `/api/sync-data` 无请求体大小限制

**文件**: `webpack.config.js`  
**严重程度**: 中

`/api/sync-data` 的 `setupMiddlewares` 中直接 `JSON.parse` 请求体，未设置 `body-parser` 的大小限制。大量笔记数据可能导致内存溢出或请求超时。

```javascript
app.post('/api/sync-data', express.json(), (req, res) => { ... });
```

### 8.4 无 TypeScript 路径别名

**文件**: `tsconfig.json`  
**严重程度**: 低

项目使用深相对路径导入（如 `../../db`、`../../contexts/ThemeContext`）。未配置 `paths` 别名（如 `@/db`、`@/components/Layout`），重构时移动文件需要更新大量导入路径。

### 8.5 生产构建不含 sourcemap 配置

**文件**: `webpack.config.js`  
**严重程度**: 低

生产构建未配置 `devtool`，出错时堆栈指向压缩后的 bundle。调试生产问题困难。
