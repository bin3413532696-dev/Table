## 2. 当前核心架构结论

### 2.1 `data/` 是文件权威源，但当前写入仍是混合语义

项目已经从“纯浏览器本地存储”切到“`webpack-dev-server` + `data/` 文件落盘”的模式，但当前代码还不是严格的 authority-first：

- `data/tasks/tasks.json` 与 `data/finance/finance.json` 是任务/财务的文件权威源
- `financeStore` / `taskStore` 仍承担前端内存态与本地缓存职责
- 用户资料、主题、PIN、API Provider 配置仍保存在 `localStorage`

当前实际规则：

1. 初始化时通过 `/api/load-data` 读取文件权威源，再与本地缓存按 `updatedAt` 合并
2. 文件变化回灌时，用服务端整包数据覆盖本地任务/财务缓存
3. 常规任务/财务 CRUD 时，当前是先 `replaceAll(...)` 更新本地 store，再异步 `POST /api/sync-data`
4. 导入与清空数据时，当前是先推送权威源，再刷新本地缓存
5. 因此现阶段更准确的表述是“文件权威源 + 本地缓存并存”，而不是“所有写操作都先写权威源”

## 5. 本轮已完成的关键改动

### 5.1 数据层切到“文件权威源 + 本地缓存”结构

已完成：

- `BaseStore.persist()` 不再自动调度旧同步引擎
- `initializeData()` 改为先读文件权威源，再与本地缓存按 `updatedAt` 合并
- `handleServerChange()` 改为服务端整包覆盖本地缓存
- `financeDB` / `taskDB` 统一收口到 `db/index.ts`，但常规 CRUD 当前仍是“本地先更新、权威源异步跟进”
- `dataManager.importAll()` / `clearAll()` 改为先写权威源，再刷新本地缓存
- `StoreContext` 旧上下文已收口到 `db/index.ts`，避免绕过权威源

注意：

- 当前 `replaceAll()` 仍会写本地缓存并触发事件
- 这在现阶段是可接受的
- 若后续目标是“常规 CRUD 也严格 authority-first”，需要调整 `runFinanceMutation()` / `runTaskMutation()` 的顺序与错误处理
- 若后续需要严格区分“权威源回灌”和“用户本地改动”，建议新增静默回灌接口，如 `replaceAll(..., { silent: true })` 或 `hydrateFromAuthority(...)`

### 5.2 智能体已支持显式工具直调，但 UI 侧仍是单轮执行

已完成：

- 当用户消息里明确点名工具名并附带合法 JSON 参数时，前端会直接构造工具调用并执行
- 常规模型输出中的 `tool` / JSON 工具块仍会被解析并执行
- 当前页面侧实际走的是 `AgentContext -> agentEngine.processMessage()` 单轮链路
- `processMessageWithAgenticLoop()` 已存在于 `AgentEngine`，但当前 UI 侧尚未接入，因此“工具结果回喂后继续多轮推理”还不是页面中的默认行为

### 5.3 增加“强行终止思考”功能

已完成：

- 在请求链路中接入 `AbortSignal`
- UI 中增加“强行终止思考”按钮
- 用户点击后，本地立即结束当前轮思考
- 已生成内容保留，并追加“已手动终止本次思考”
- 不清空历史消息
- 流式读取时会显式取消 `reader`

### 5.4 任务/财务已开始自动映射进知识库

已完成：

- `src/db/index.ts` 已接入任务与财务到知识库的后台同步映射
- 任务新增、编辑、切换完成状态、删除后，会自动 upsert 或清理对应 `class:task` 实体
- 财务新增、编辑、删除后，会自动 upsert 或清理对应 `class:finance-record` 实体
- 映射实体 ID 采用稳定规则：
  - 任务：`entity:task-${task.id}`
  - 财务：`entity:finance-${record.id}`
- 映射关系已接到 `entity:workspace`
  - 任务使用 `relation:linkedTask`
  - 财务使用 `relation:linkedFinanceRecord`
- 导入与清空数据后，会重建或清理这批模块映射实体，避免知识库残留脏映射

注意：

- 当前仍沿用任务/财务模块既有的 `local-first + async push` 主流程
- 知识库映射失败不会阻断任务/财务主操作，只会在控制台输出 warning
- 这属于渐进式接入，不是严格事务一致的双写方案

## 10. 本体知识库接入计划

### 10.1 目标定位

计划在当前项目中新增一层“本体知识库”，用于增强智能体对项目内知识、实体关系、来源证据和跨模块上下文的理解能力。

目标不是一开始就在浏览器里实现完整语义网推理器，而是优先落地：

- 可建模：能定义实体类型、属性、关系和来源
- 可检索：能按关键词、标签、类型、关系快速找到知识
- 可引用：智能体回答时能引用实体、关系和来源
- 可同步：知识数据能沿用当前 `data/` 文件权威源机制落盘和回灌

### 10.2 与当前项目架构的适配原则

结合当前仓库结构，建议遵循以下原则：

1. 文件权威源不变  
   知识库数据仍落到 `data/` 目录，不引入额外服务端数据库。

2. 前端本地索引单独建设  
   不复用 `taskStore` / `financeStore` 的数据结构，知识库应独立建模块。

3. 智能体优先走工具，不先重写对话引擎  
   先把知识库能力接成工具，挂到现有 `toolRegistry`，再决定是否切到多轮 agentic loop。

4. 本体建模轻量化  
   采用“OWL 风格建模 + JSON-LD 表达 + 轻规则推理”，不在第一阶段追求完整 OWL DL 推理。

5. 先结构化知识，后向量检索  
   先做词法检索、关系扩展、来源追踪；Embedding / Hybrid Retrieval 放到后续增强阶段。

### 10.3 推荐技术路线

推荐采用以下分层：

- 本体/交换格式：`JSON-LD`
- 浏览器本地索引：`Dexie`
- 前端词法检索：`Fuse.js`
- 可选 RDF/Turtle 互通：`N3.js`
- 可选 JSON-LD 处理：`jsonld.js`

原因：

- 当前项目本身就是 JSON 文件权威源，`JSON-LD` 与现有落盘方式兼容性最好
- `Dexie` 已在依赖中存在，但当前尚未实际启用，适合作为知识库索引层
- `Fuse.js` 已在依赖中存在，适合做 MVP 阶段的本地全文/模糊检索
- 后续若需要和外部 RDF/图谱生态互通，可再引入 `N3.js` / `jsonld.js`

### 10.6 前端模块划分建议

建议新增独立知识库模块，而不是继续堆到 `db/` 或 `store/impl/` 里：

```text
src/
  kb/
    types.ts
    schema.ts
    serializer.ts
    store.ts
    indexer.ts
    query.ts
    rules.ts
    sync.ts
```

职责建议：

- `types.ts`
  定义 `OntologyClass`、`Entity`、`Assertion`、`DocumentNode` 等 TS 类型
- `schema.ts`
  管理本体元数据、枚举、字段规范
- `serializer.ts`
  负责 TS 对象与 JSON-LD 的双向转换
- `store.ts`
  负责运行时数据访问和本地内存态
- `indexer.ts`
  建立 `Dexie` 索引和 `Fuse.js` 检索索引
- `query.ts`
  负责搜索、过滤、关系扩展、上下文压缩
- `rules.ts`
  放轻量规则推理逻辑
- `sync.ts`
  对接现有 `/api/load-data`、`/api/sync-data`

### 10.7 推荐的知识库读写语义

建议知识库沿用当前项目“文件权威源 + 前端缓存”的总体思路，但不要完全照搬现有任务/财务的 local-first 异步写回。

建议知识库采用：

- 读：初始化时从 `/api/load-data` 获取知识数据，再建立本地索引
- 写：先构造完整下一状态，再同步到 `data/knowledge/*.jsonld`
- 缓存：`Dexie` 只做浏览器索引缓存，不作为最终权威源
- 回灌：文件变化时整包刷新知识图运行态与搜索索引

如果后续要避免当前任务/财务模块那种“local-first + async push”语义混淆，知识库模块应从一开始就优先靠近 strict authority-first。

### 10.8 轻量推理范围建议

第一阶段只建议做以下几类规则：

- `subClassOf` 继承展开
- `inverseOf` 反向关系补全
- `sameAs` / `aliases` 别名归并
- `partOf` / `dependsOn` 的有限深度传递展开
- `mentions -> relatedTo` 的弱关联补全

第一阶段不建议做：

- 完整 OWL DL 推理
- 浏览器内复杂一致性验证
- 通用 SPARQL 引擎
- 大规模图数据库式查询优化

原因：

- 当前项目首先需要的是“Agent 可用、可解释、可维护”
- 不是“学术级本体推理平台”

### 10.10 检索策略建议

1. 词法检索
2. 图关系扩展

## 12. 知识库收敛优化计划

### 12.1 问题判断

当前知识库能力已经具备较完整的底层结构，但从产品形态看，开始出现“能力过多、层次混合、主流程不够清晰”的问题。

当前杂乱感主要来自以下几类能力混在同一处：

- 本体层：类、关系、断言、文档证据
- 业务层：任务、财务自动映射
- 维护层：新增、编辑、删除、关系补录
- 调试层：authority 落盘、结构校验、回归验证

结论：

- 从技术实现角度，不算无意义堆砌
- 从页面产品形态角度，已经开始偏“图谱控制台”
- 若目标是服务智能体与日常项目使用，知识库应收敛为“知识中心”，而不是继续朝“通用本体编辑器”扩张

### 12.2 优化目标

下一阶段优化目标应收敛为：

- 让知识库主场景聚焦在“查找知识、理解上下文、追溯来源”
- 保留现有底层能力，但不再默认全部暴露
- 明确区分自动映射数据与人工维护数据
- 把高级图谱编辑能力迁移到“高级模式”或“维护模式”
- 让智能体优先消费低噪音、可追溯、结构稳定的知识视图

### 12.3 优化原则

建议按以下原则收敛：

1. 检索优先  
   默认入口先服务搜索、浏览、理解，而不是先服务建模与编辑。

2. 分层暴露  
   把查看、维护、高级结构编辑拆成不同层，不让所有能力同时出现在主界面。

3. 来源优先  
   详情页优先展示“这是什么、有什么依据、关联什么业务对象”，而不是优先展示底层三元组结构。

4. 自动映射与人工知识分层  
   任务、财务自动映射应作为基础上下文层展示，不与人工知识录入完全混排。

5. 保能力、不保噪音  
   底层能力可以保留，但默认隐藏到高级模式，减少普通路径的信息密度。

### 12.4 详细执行计划

#### 阶段 A：产品范围收敛

目标：

- 明确知识库第一阶段产品定位
- 暂停继续增加新的底层图谱编辑入口

动作：

- 把知识库重新定义为“项目知识中心”
- 冻结第一阶段范围：保留检索、浏览、来源追踪、任务/财务自动映射、少量人工补充
- 暂停继续扩展默认可见的本体编辑、复杂断言编辑、通用关系维护入口

产出：

- 一版收敛后的功能边界说明
- 一版主界面与高级模式的职责划分

#### 阶段 B：信息架构拆层

目标：

- 将当前混合式页面拆成更清晰的使用层级

建议拆分为三层：

1. 知识中心  
   默认视图，仅包含搜索、筛选、结果列表、详情阅读、来源查看、关联任务/财务查看

2. 维护模式  
   提供高频但低复杂度的维护能力，如标题、摘要、标签、来源文档、简单关联补充

3. 高级模式  
   放置原始断言、关系边、本体类、关系定义、调试信息、原始结构查看

产出：

- 知识库页面结构改版图
- 默认模式与高级模式切换方案

#### 阶段 C：首页改成检索优先

目标：

- 把知识库首页从“结构化编辑入口”改成“搜索与阅读入口”

动作：

- 首屏优先展示搜索框、最近更新、重点实体、来源文档、业务映射入口
- 弱化“新建实体 / 新建文档 / 新建断言 / 新建关系”的直接入口
- 将创建入口从首屏主操作改为次级入口

产出：

- 新首页布局

#### 阶段 D：主界面字段瘦身

目标：

- 降低普通用户面对的结构复杂度

动作：

- 别名、置信度、属性 JSON、原始 source、结构属性默认折叠
- 不再在普通模式中显式突出 `typeId`、`predicateId`、`objectId` 等底层字段

- 折叠区与高级字段展示规则

#### 阶段 E：自动映射与人工知识分区

目标：

- 解决任务/财务映射与人工知识混杂的问题

动作：

- 在 UI 中显式标记“系统生成”与“人工维护”
- 搜索与筛选中增加按来源、按维护方式过滤
- 在详情页中区分：
  - 业务映射信息
  - 知识摘要
  - 来源文档
  - 高级结构

产出：

- 自动映射实体视觉标识
- 人工知识与映射知识的筛选器

#### 阶段 F：高级能力后移

动作：

- 通用关系维护改为从实体详情中的“补充关联”进入
- 本体类与关系定义维护只保留在开发者或高级模式

目标：

- 提升知识库对智能体的可用性，减少噪音

动作：

- 为智能体场景优先整理以下字段：
  - 摘要
  - 来源
  - 最近更新时间
  - 关联任务
  - 关联财务
  - 关键文档
- 避免让智能体默认依赖大量低质量手工属性或底层结构字段

产出：

- 一版“智能体优先视图”定义
- 智能体检索与展示字段白名单

#### 阶段 I：术语与文案收敛

目标：

- 降低非图谱使用者的理解门槛

动作：

- 普通界面少用“实体、关系、断言、本体”等术语
- 改用更贴近日常理解的表达：
  - 知识条目
  - 关联
  - 来源
  - 说明文档
- 将专业术语保留在高级模式

产出：

- 一版普通模式文案表
- 一版高级模式术语保留表

#### 阶段 J：验证与验收

目标：

- 用真实任务验证知识库是否从“功能多”变成“主流程清晰”

建议验收任务：

1. 能否在 10 秒内找到某个任务关联的知识
2. 能否快速理解某条财务记录的上下文与来源
3. 能否让智能体基于来源文档回答问题
4. 能否区分系统自动映射与人工维护内容
5. 能否在不进入高级模式的情况下完成日常查看与补充

验收标准：

- 普通用户首屏不会被结构化编辑表单淹没
- 自动映射数据和人工知识可以一眼区分
- 高级结构能力仍可用，但默认隐藏
- 智能体读取到的知识更加稳定、低噪音、可追溯

### 12.5 建议执行顺序

建议按以下节奏推进：

1. 第 1 阶段，1 到 2 天  
   只做方案收敛、页面结构设计、功能分层，不扩新功能

2. 第 2 阶段，2 到 3 天  
   改知识库首页和详情页，完成“检索优先”重排，隐藏高级编辑入口

3. 第 3 阶段，1 到 2 天  
   增加来源分区、自动映射标识、筛选器、折叠字段

4. 第 4 阶段，1 到 2 天  
   建立高级模式与调试抽屉，迁移现有底层能力

5. 第 5 阶段，1 天  
   做一次智能体联调与人工走查，确认主流程清晰度提升

### 12.5.1 当前已落地进展

截至 2026-05-04，以下收敛项已经开始落地到页面实现：

1. 阶段 A：产品范围收敛

- 已明确将知识库页面定位收敛为“项目知识中心”
- 默认文案已从“图谱维护页”转向“检索、阅读、来源追踪优先”

2. 阶段 B：信息架构拆层

- [src/pages/Knowledge/index.tsx](D:/project/Table/src/pages/Knowledge/index.tsx) 已增加三层模式：
  - 浏览模式
  - 维护模式
  - 高级模式
- 默认模式已切到浏览模式
- 新增、删除、断言、关系等高噪音操作不再默认全部暴露

3. 阶段 C：首页改成检索优先

- 已新增“知识首页”区域，提供任务映射、财务映射、人工知识/来源文档等快速入口
- 已新增“推荐阅读”与“最近更新”区域
- 默认首屏强调查找、阅读与上下文理解，而不是先展示结构化编辑入口

4. 阶段 D：主界面字段瘦身

- 实体详情已优先展示“关键信息”“来源文档”“关联实体”
- 任务/财务自动映射实体与人工维护实体已显式区分
- 扩展属性、直接关系、断言等低频结构信息已改为按模式展示，并进一步收进折叠块

5. 阶段 F：高级能力后移

- 高级模式中已新增“调试信息”独立入口
- 已将查询状态、当前选中节点、结构统计等信息收进独立调试弹层
- 已新增 authority 文件级诊断，可查看 `data/knowledge/*.jsonld` 的存在状态、条目数、体积与修改时间
- 调试信息不再继续堆在主浏览区内

6. 阶段 G：新增入口重做

- 维护模式下已将默认新增入口收敛为“补充知识”
- 新增入口已改成任务导向的轻量向导，而不是直接暴露“新增实体 / 新增文档 / 新增关系”等术语
- 当前向导已支持：
  - 补充项目知识
  - 补充说明文档
  - 补充业务备注
  - 补充结构关联

当前仍未完成的部分：

- 调试抽屉已补到原始节点 JSON、结构快照摘要、服务端 `/api/load-data` 快照对比与 authority 文件级状态，但尚未做到文件内容级差异诊断
- 新增入口已开始按“项目知识 / 说明文档 / 业务备注”做独立轻量表单，但高级维护路径仍保留通用全量表单
- 普通模式 / 高级模式的文案分层已经开始落地，普通阅读路径已优先改用“知识条目 / 来源文档 / 事实记录 / 结构关联”等更任务化表述，但仍需继续补齐剩余高级区与调试区文案
- 还未做针对这轮收敛后的人工走查结论沉淀

### 12.6 目前不建议继续做的事

在完成本轮收敛之前，不建议继续默认暴露以下能力：

- 更多本体类与关系定义编辑入口
- 更多原始断言级别的主界面操作
- 更多面向调试的底层结构信息
- 把更多业务模块直接映射进知识库主界面而不先做分层

### 12.7 本阶段优化后的目标形态

优化完成后，知识库应更接近以下形态：

- 对普通用户：是“项目知识中心”
- 对维护者：有“维护模式”
- 对开发者：有“高级模式 / 调试模式”
- 对智能体：是“低噪音、可追溯、可引用的上下文源”

也就是说，下一阶段的目标不是继续把知识库做得更“大”，而是把它做得更“清晰、克制、可用”。
