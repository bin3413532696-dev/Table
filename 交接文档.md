## 2. 当前核心架构结论

截至 2026 年 5 月 5 日，项目已经完成从“前端本地存储主导”到“后端 PostgreSQL 权威”的关键切换。

当前必须按以下结论理解项目：

1. `tasks` 的权威源是 `PostgreSQL`
2. `finance` 的权威源是 `PostgreSQL`
3. `knowledge` 的权威源是 `PostgreSQL`
4. 前端不再是主业务真相写入者
5. `data/` 目录不再承担任务、财务、知识库的业务权威职责
6. `tasks / finance -> knowledge` 已改为后端 `outbox + projection runtime` 异步投影
7. ontology 已独立拆表，不再以 `dataset_json` 作为权威源
8. 前端知识刷新已改为基于 `/api/knowledge/metadata` 的轮询刷新，不再依赖无效 `/ws`

一句话总结：

当前项目的正确理解方式是：

- 后端权威已建立
- 知识库结构化已建立
- 异步投影底座已建立
- 用户体系、智能体服务端化、搜索层仍未开始

## 3. 当前已完成改造

### 3.1 业务主数据后端权威化

已完成：

- 新增独立后端目录 `server/`
- 接入 `PostgreSQL + Prisma`
- `tasks` 与 `finance` 已切换到后端 API 驱动
- 前端 `taskDB` / `financeDB` 已改为调用服务端
- 开发期已具备 `reset / snapshot / seed` 基础能力

### 3.2 知识库后端权威化

已完成：

- `knowledge` 已切换到后端 API + PostgreSQL 权威模型
- 已提供：
  - `GET /api/load-data`
  - `POST /api/sync-data`
  - `GET /api/knowledge/dataset`
  - `PUT /api/knowledge/dataset`
  - `GET /api/knowledge/metadata`
- 前端知识库导入导出、初始化、刷新链路已对接后端知识接口

### 3.3 知识库存储结构化拆表

已完成：

- `knowledge_bases` 退化为 metadata 容器
- 已落地知识表：
  - `knowledge_entities`
  - `knowledge_relations`
  - `knowledge_documents`
  - `knowledge_document_entities`
  - `knowledge_assertions`
  - `knowledge_assertion_evidence`
- 已落地 ontology 表：
  - `knowledge_ontology_classes`
  - `knowledge_ontology_relations`

当前语义：

- 结构化表是知识库真实写入位置
- `/api/load-data` 与 `/api/knowledge/dataset` 是后端组装视图
- `replaceKnowledgeDataset()` 仅保留给导入、重置、恢复等全量替换场景
- 常规 CRUD 已切换为结构化增量写入

### 3.4 任务/财务到知识库的异步投影

已完成：

- 新增 `projection_outbox_events`
- `tasks` 与 `finance` 写入改为“业务写入 + outbox 入队”
- 新增 `server/src/modules/projection/`
- 后端 runtime 会异步消费任务/财务事件并写入知识投影
- 前端 [src/db/index.ts](D:/project/Table/src/db/index.ts) 中原有直接双写知识库逻辑已移除

### 3.5 ontology 独立表第一版

已完成：

- 新增 `knowledge_ontology_classes`
- 新增 `knowledge_ontology_relations`
- `knowledgeBase.datasetJson` 不再作为 ontology 权威源
- `repository.ts` 已优先从独立表组装 ontology
- `seed / reset` 已同步处理 ontology 独立表

### 3.6 前端知识刷新链路收口

已完成：

- 新增 `GET /api/knowledge/metadata`
- 前端移除对 `/ws` 的无效依赖
- 改为 metadata 轮询刷新

## 9. 当前主要风险与边界

### 9.1 默认用户仍是硬编码占位

风险：

- 没有真实用户上下文
- 无法做权限、隔离、审计

### 9.2 智能体仍以前端链路为主

风险：

- 缺少正式 run 生命周期
- 缺少服务端恢复与审计
- 缺少统一工具执行边界

### 9.3 搜索层尚未建立

风险：

- 知识、任务、财务虽然已结构化，但尚未形成正式检索能力

### 9.4 目录仍有过渡期痕迹

现状：

- 根目录仍保留若干开发日志文件
- `data/` 目录仍作为历史调试产物存在

结论：

- 当前不建议立即做大规模物理搬迁
- 应先继续稳住架构边界，再做目录收敛

---

## 10. 接手后的推荐推进顺序

建议按以下顺序继续：

1. 建立真实用户体系、鉴权中间件与用户上下文注入
2. 为 `knowledge` 增加 ontology 编辑 API、约束校验与重建入口
3. 建立正式搜索层，先落 PostgreSQL FTS
4. 将智能体改造成服务端 `run / message / tool_execution` 状态机
5. 再考虑向量检索、审计、权限细化与目录大整理

如果继续做代码，而不是继续写文档，最合理的下一步是：

- `ontology` 后端编辑 API
- 搜索索引层
- 智能体服务端化
- 用户体系 / 鉴权 / 租户隔离
